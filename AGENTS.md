# SpaceMolt Multi-Agent Harness

Canonical agent context file. CLAUDE.md imports this file — edit this one only. Claude Code (terminal) is the primary tool for this project; the context system is plain markdown in git, so it needs no per-vendor translation to remain portable.

## What this project is

A TypeScript/Bun harness that runs 3 LLM-powered agents playing SpaceMolt (an MMO built for AI agents, https://spacemolt.com), with an embedded web dashboard for monitoring and steering them. The game is the sandbox; the real goal is learning to lead AI agent teams, engineer loops, and build harnesses. Cost minimization is a first-class design goal: LLM calls via Claude subscription (headless) and local Ollama, plan-then-execute so the LLM plans rarely and deterministic code executes.

## Working conventions (binding)

- **Update `docs/STATE.md`** after completing any task and at session end. It is the handoff; a fresh session in any tool must be able to continue from it alone. Its top **`## NOW` block is PM-owned and must be refreshed at every wave of work, every merge cluster, and every compaction/away-transition — including IN-FLIGHT work (dispatched-but-unmerged) — so remote progress is visible without reading code. STATE.md is a living handoff with no logic to review; refresh it via a lightweight self-merged docs PR rather than letting it move only when batches merge.
- **Maintain `README.md`** as the overall project summary (game + learning aspect + key decisions + progress checklist). Update its Progress section when milestones land; keep it a summary that links out — decisions.md and STATE.md hold the detail (SSOT).
- **Append `docs/decisions.md`** whenever a significant choice is made, written for an infrastructure/automation engineer, not a developer. Each entry must show the decision as it was actually made: (1) the problem/context, (2) the **options that were on the table** — each with what the analysis or debate showed for it and its tradeoffs, (3) the **decision and why** it beat the alternatives. If only one option was genuinely considered, say so plainly (don't invent a fake debate). The goal: a reader sees not just what we chose but what we rejected and why, so any future reversal is an informed one. **Hard shape and cap (enforced by `test/doc-size.test.ts`): 400 words per entry, one-paragraph context, `**Options.**` as terse bullets (one line each — the option, its tradeoff, the verdict), one-paragraph `**Decision.**`.** Typical entry is 200-300 words; 400 is the ceiling, not the target. Supporting detail goes in the PR or issue the entry cites — the log is an INDEX of decisions with their reasons, not a narrative, and a reader who wants the story follows the link.
- **Educational register in all documentation prose** (wiki, decisions, README, STATE): teach as you inform. The reader is moderately tech-literate but not an expert — comfortable with computers and general concepts, not fluent in developer or AI jargon. Define terms in plain words on first use, build from first principles (what problem does this solve?) before the specifics, prefer a concrete analogy to an abstract definition, and give the why alongside the what. Specs, plans, code comments, and inter-agent traffic are exempt (working engineering artifacts stay precise and terse).
- **Brevity is part of the register — teach in a paragraph, not a page.** "Educational" licenses explaining the why; it does not license an essay. The register drifted into novel-length prose and the operator called it: *"right now we are on track to have a full novel to read through and that is too much."* So the caps are mechanical and executable (`test/doc-size.test.ts`) — and they cap the UNIT OF WORK, not the accumulated file: 400 words per NEW decisions entry, 500 words for the STATE `## NOW` block. Whole-file and whole-wiki-page byte caps are deliberately not gated (operator ruling, 2026-07-14: a cap per update beats a hard gate on the whole file); whole-file growth is a doc-steward archival trigger instead, and `engineering-lessons.md` is exempt from all of it — it IS the curriculum, and exempt means uncapped, not unedited. Shorter, not dumber: the cap is a brevity gate, never a content gate. It takes narrative, repetition, and self-congratulation. It never takes a fact, a rejected option, a receipt, or a design detail — an entry that cannot fit those in 400 words keeps them and is grandfathered on the test's legacy list. A doc nobody re-reads teaches nobody.
- **Run `prose-lint` on documentation prose before opening a PR.** The Vale-based `prose-lint` skill (ai-tells + Cringely styles) is a deterministic check — no new dependencies, no tokens. Any agent producing or editing wiki/decisions/README/STATE prose runs it on the files it touched and fixes what it flags; a genuine false positive gets named in the PR body rather than silently ignored. It is in the doc-steward charter as step 8.
- **Jargon is a tax on the reader; pay it once, then stop.** Our house vocabulary (ablate, forcing function, invariant, seam, producer/consumer, net-negative) is precise, and repeated to exhaustion it stops being read at all. Rules for anything a human reads (wiki, decisions, README, STATE, chat, PR bodies): use the plain sentence where it does the same work ("delete the fix and confirm the test fails" beats the one-word version); gloss a term of art in place, in six words, the first time it appears; and don't let one word carry three paragraphs. Recycled sentence frames fatigue a reader the same way a repeated noun does. The only mechanical backstop is `Cringely.AblationOveruse`, which warns when a single PARAGRAPH leans on that word three or more times. Vale's occurrence check counts per paragraph and cannot aggregate per document, so a term that recurs at a steady drip across a whole file is a human call, not a gate. A warning is a question, not a verdict: a lessons entry about the technique may exceed it and say why. EXEMPT: agent briefs, review verdicts, code comments, specs and plans, where a fluent audience gains precision from the term of art and the plain paraphrase is the thing that costs.
- **Add `docs/wiki/` pages** when durable knowledge is learned (game mechanics, API quirks, operational lessons). One topic per page, small pages, link from the table above or STATE.md.
- **Documentation-freshness gate (binding).** Freshness is part of the definition-of-done for a merge cluster, not an afterthought. After any PR merges to main, a doc-steward pass (see `docs/wiki/team-structure.md`, the Doc Steward role — a dedicated agent the PM dispatches after each merge cluster) reconciles the living docs against reality BEFORE the next batch dispatches: the STATE.md `## NOW` block, `docs/milestones.md` (and the PM then republishes the claude.ai milestone-tracker Artifact from it, since this file is that Artifact's SSOT), the README progress section, `docs/wiki/engineering-lessons.md` when a merge taught a transferable lesson, and any decisions.md cross-references the change created. Promoted to a binding rule because doc-staleness recurred (STATE.md lagged behind batch merges; the milestone tracker went stale despite ~20 merged PRs) — a recurring failure class becomes a standing constraint. The steward's remit is the operator's remote view: if a person reading only these docs would be misled about where the project stands, the merge cluster is not done.
- **Teach the principle — the curriculum never drifts behind the work.** Every significant step, phase, or decision captures the educational "why it matters": the general principle it demonstrates and which discipline it belongs to (AI / harness / loop engineering). When that principle transfers beyond this project, append it as a lesson to `docs/wiki/engineering-lessons.md` (concrete incident → principle → discipline → why, per the template there); don't duplicate the full story, cross-link to the decisions.md entry. The project's real purpose is learning, so a lesson left uncaptured is work half-done.
- **Team-structured execution** (see `docs/wiki/team-structure.md`): the main context is the PM; tech-lead agents run task batches with ephemeral implementer + independent reviewer agents; council reviews at milestone gates only.
- **Completion reports use the five-field template** (`docs/wiki/team-workflow.md`): SHIPPED / EVIDENCE / FINDINGS / BLOCKERS / FILES, capped at 150 words EXCLUDING evidence lines (test counts, ablation results, file:line cites — those never compress). The dispatching seat pastes the empty template into every brief and bounces a prose report with one word: "reformat". The old telegraphic-traffic rule was prose, and prose did not hold — reports still ran 300-800 words. Not compressed by this: dispatch briefs (compression causes intent drift), reviewer PR verdicts (they outlive the reviewer and the ADVANCE bar needs receipts), and human-facing artifacts.
- **Independent review, never self-review**: specs, plans, and code are reviewed by a fresh agent/context that didn't author them.
- **SSOT/DRY/KISS**: every game action is defined once in `src/registry/actions.ts`; the OpenAPI spec is the SSOT for the game's data model; no speculative abstraction.
- **Check the vendored reference before you guess (binding).** Anything that touches a game tool, request/response shape, item, or mechanic gets checked against `docs/game-reference/` first — by the implementer AND by the reviewer. **Evidence precedence: (1) a live capture from the game beats (2) the vendored reference, which beats (3) any assumption.** The vendored reference is authoritative until reality contradicts it — at which point reality wins and the contradiction gets captured as a fixture and noted in the reference with a dated correction. This project has been burned both ways: the game's own refuel prose taught `fuel_cells` when the real item id is `fuel_cell` (#179), and the `auto_list` behavior the docs implied was falsified live (2026-07-12). Never guess; prefer the reference over your instinct; prefer a capture over the reference. An implementer who cannot cite the reference (or a live capture) for a shape they relied on has not finished; a reviewer who approves an uncited shape has not reviewed. The references are static and vendored, so checking costs nothing — that is the point. Live probing stays the authorized Batch-0 pattern for genuinely NEW data only, and what it learns is written back here. (Promoted after the same failure recurred: `auto_list` assumed, palladium assumed worthless, `view_market`/`get_system` shapes guessed — all of it published upstream the whole time. Issue #105.)
- **Simplicity rules are binding** (`docs/wiki/simplicity-rules.md`, imported from an earlier project's postmortem): fix the producer not the consumer, isolate before bundling, complexity needs a receipt, review verdict is correct-AND-smallest (correct-but-larger = revise), enumerate every cache input, no load-bearing unknowns in plans.
- **Secrets** live in `secrets/` (gitignored) — never in code, config, logs, or commits. Registration code comes from https://spacemolt.com/dashboard.
- **Commits**: user's identity only (no AI co-author trailers). Same for PR bodies, issues, and comments: no "Generated with Claude Code" footer or any AI attribution (operator order 2026-07-16, overrides tool defaults). Commit and push working state before ending a session.
- **Git workflow — main is protected.** No direct commits or pushes to main; every change lands via a pull request (batch branches like `batch/c-store-executor`, doc branches like `docs/git-workflow`). Merge with `gh pr merge` after the batch report/review, then pull main. Enforced by the versioned pre-push hook in `.githooks/` — one-time setup per clone: `git config core.hooksPath .githooks`. (Server-side protection needs GitHub Pro on private repos, so the hook is the enforcement.)
- **Tests are offline**: fake server + mocked planner, zero live-game traffic, zero LLM tokens. `bun test && bun run typecheck` must pass before any commit claiming a task done.
- **No live calls without authorization**: no agent (implementer, reviewer, researcher) makes live LLM calls or live-game calls unless its brief explicitly authorizes it. Applies to "quick probes" and research too. Security baseline (`docs/wiki/security-baseline.md`) is binding.
- **Invariant promotion**: a failure class appearing twice gets promoted to a permanent constraint — deterministic code for agent/pilot failures, a binding convention here or in the wiki for process failures. The incident record then shrinks to a pointer. A promoted constraint that stops earning its line gets pruned.
- **Deterministic lessons feed improv mode**: every deterministic guard/normalizer/lesson we add (a wake suppression, a normalizer, an effect-check, a channel enum) also gets a paired **improv-mode instruction** recorded in the improv-mode spec (`docs/superpowers/specs/2026-07-12-improv-mode.md`) — either as a prose briefing rule or a note that it's a non-negotiable backstop that stays deterministic. Our deterministic code is crystallized lessons; a self-driving agent loses them unless captured. The improv briefing must never drift behind the code.
- **Persisted-state schema tolerance**: any schema tightening (a stricter enum, a new required field) must be tested against a stored artifact that PREDATES it, and loaders must DISCARD an artifact that no longer validates rather than crash — persisted state (plans, goals, events) outlives the schema that wrote it. (Promoted from the 2026-07-12 chat-enum incident: a stricter `chat.target` enum crash-looped production by rejecting a stored plan.)
- **A test pilot stays online** for continuous training and tuning; experiments are annotated in agents.yaml (hypothesis, date, revert condition) and results land in the decision log.
- **Value density over volume — no busywork.** Counts of tests, review findings, or documents are not productivity metrics and are never a goal. A test earns its place only if it fails when real behavior breaks: distinct failure mode, behavior not implementation, no restating the code, no padding. 15 high-value tests beat 50 where 30 prove nothing. Same for reviews: report zero findings when there are zero; manufactured nits are worse than silence because they bury real defects. When writing plans/specs/tests, ask of each item "what breakage does this catch?" — no answer, no item.

## Stack

Bun ≥ 1.2.21, TypeScript, Zod (only runtime dependency). SQLite via `bun:sqlite` for events + plan cursors. React/Vite dashboard (Plan 3). Container-first deployment (Plan 4). Game protocol: HTTP API v2 (`https://game.spacemolt.com/api/v2/{tool}/{action}`, `X-Session-Id` header).
