## 2026-07-13 — Checking mining preconditions deterministically (equipment first, deposit-match briefed)

**Problem — a pilot tried to MINE with no mining laser fitted.** This is the same failure class we have now hit several times: attempt an action without checking its preconditions. The undock-when-not-docked guard closed it for undock; the station-awareness `[station]`/dock briefing closed it for dock; the accept_mission template_id guard closed it for missions; here it is mine. The lesson those taught is the frame for this one: preconditions are CHECKED deterministically at the moment of acting, not remembered by the planner. A pilot that "knows" it needs a laser still forgets under a long plan, a cheap-tier model, or an improv window; a harness check does not forget. Mining has a hard, cheap-to-check precondition — a mining laser module has to be fitted — and the fitted-modules list is already sitting in every `get_status` response we fetch each tick.

**What the available data supports (and what it doesn't).** The live probe fixture (`test/fixtures/spacemolt-probe-2026-07-12.json`) shows `get_status.structuredContent.modules[]` as a top-level sibling of ship/player/location: each entry carries `type` ("mining"), `type_id` ("mining_laser_i"), and `stats.mining_power` (5). That is enough to answer the cheap question — is ANY mining laser fitted — with certainty. It is NOT enough to answer the fuller question the miner's-guide rule raises (does your mining_power suit THIS deposit): a deposit's `supported_power` lives in the `get_poi` response, whose shape is still ASSUMED/uncaptured (get_poi is a near-dead path; only get_system and find_route were captured live). We refused to build the deposit-support pre-check against a guessed get_poi shape — that would be a load-bearing unknown, exactly what the simplicity rules forbid.

**Options.**
- *(A) Deterministic fitted-laser guard + grounded briefing [SHIPPED].* Parse `modules[]` onto `StatusSnapshot.modules` (VERIFIED shape), and before a `mine` action short-circuit to a `blocked` wake when the fitted set is KNOWN and contains no mining laser — mirroring the undock/inTransit guard pattern, no doomed call sent. Pair it with a planner briefing (grounded in the catalog's mining_laser_i..v equipment at `src/catalog/catalog.data.json`, loaded via `src/catalog/catalog.ts`, and the miner's-guide rule) that states the equipment requirement AND the deposit-matching rule: a deposit has a `supported_power`, and total mining_power beyond ~4x it CANNOT lock the deposit — a bigger laser makes sparse/depleted deposits WORSE, so match the tool and move to fresh veins rather than scraping. Cheap, certain, and it teaches the counter-intuitive part (bigger is not better) the planner would otherwise get wrong.
- *(B) Also build the deposit-support pre-check now [DEFERRED].* Compare live mining_power to THIS deposit's supported_power and block a doomed lock before it is attempted. Deferred, not rejected: it needs the `get_poi` response shape, which we have never captured. Building it now means guessing that shape, and a wrong guess flies the pilot blind exactly as the pre-fix get_poi/find_route guesses did (see the SM-2 diagnosis). It is the natural follow-up the moment a live get_poi is captured; until then the briefing carries the rule as guidance and the pilot learns a sparse-deposit block the ordinary way (one blocked wake, then relocate).
- *(C) Briefing only, no deterministic guard [REJECTED].* Just tell the planner it needs a laser. Rejected because a briefing is a soft nudge a cheap model or an improv agent can ignore, and the whole point of this failure class is that the un-missable check belongs in deterministic code — the same reason undock and dock got guards, not just prose. The briefing is the paired improv-mode instruction, not the primary control.

**Decision — ship A.** It is the smallest change that makes the guaranteed-failure case un-missable using data we already fetch, and it stays honest about the boundary: the cheap check that available data supports is deterministic; the fuller deposit-support check that needs an uncaptured shape is a named, deferred follow-up rather than a guess. The guard fires ONLY when modules are KNOWN (a present array); an absent or malformed modules block maps to `undefined` (UNKNOWN) and the guard fails safe — skipping, so a shape surprise never fabricates a "no equipment" block from missing data, and `classifyGameError` still catches any real block at the call site. This is the inverse default of the cargo manifest's `.catch([])`: for a guard that short-circuits an action, "unknown" is the safe default; "empty" is only safe for a manifest that merely informs.

**Improv-mode note.** Per the deterministic-lessons-feed-improv rule, the guard has a paired briefing rule recorded in `docs/superpowers/specs/2026-07-12-improv-mode.md` (the mining requirement + the deposit-match rule) and is listed among the deterministic backstops that REMAIN — a self-driving agent gets the guidance in prose, but the harness still blocks a laser-less mine whether or not the model heeds it.

**Provenance.** Reimplements the reviewed-but-unmerged PR #127 (closed after a rebase tangled its tests) cleanly on current main, alongside the existing undock and accept_mission guards.

## 2026-07-13 — The mission funnel was structurally dead: the harness fetches the listing, the planner only accepts and completes (#147)

**Problem — the digest instructed a step the validator was guaranteed to reject.** The mission briefing (from #124) told the planner: run `get_missions`, read a template_id, then `accept_mission`. But `get_missions` is registered as a *query* (a read-only lookup), and plans are deliberately made of *mutations* only — actions that change the world. PlanSchema, the gate every plan passes through, rejects any step that isn't a mutation. So the prompt commanded a step the gate always refused. The event store showed the consequence plainly: 11 planner errors in 48 hours from plans carrying `get_missions`, 4 empty `accept_mission` attempts (the planner skipping the listing it couldn't fetch — correctly blocked by the executor guard from the 2026-07-12 fix), and zero mission steps ever executed. The one `missions_completed=1` on the books was a server-side auto-completing starter objective, not our flow. The violated invariant, in one sentence: every action the digest instructs the planner to plan must be admissible by PlanSchema.

**Options.**
- *(A) Harness fetches the listing; the digest carries the raw text [SHIPPED].* When the pilot is docked, the agent calls `get_missions` itself — a free query in the same family as the status and surroundings fetches it already makes each replan — and appends the raw listing text to the digest, quoted and truncated like every other piece of untrusted game text. The briefing then says "the missions at this station are listed above; plan accept_mission with a template_id copied from that listing" — one plannable mutation step, no schema change. The response shape stays unparsed on purpose: we have never captured a live `get_missions` response, and guessing shapes is exactly what flew earlier flights blind (the SM-2 lesson). Raw pass-through lets the planner read template_ids directly off the text.
- *(B) Admit query steps into PlanSchema [REJECTED].* Let plans contain `get_missions` so the instructed sequence becomes plannable as written. Rejected because a query step is only useful if a LATER step can consume its result — which needs a mid-plan result channel (executor feeds step N's output into step N+1's params) that does not exist. Building that channel for one listing is a structural change to the plan/execute contract; the whole design premise is that plans are linear mutations and anything needing a mid-plan decision ends the plan. Correct-but-larger, so revise.
- *(C) Briefing-only rewording, no fetch [REJECTED].* Just stop naming get_missions and tell the planner to accept missions "when it knows one". Rejected: without the listing the planner has no template_id to copy, so this converts the visible planner_error class into invisible guessing — the exact empty-param accept the executor guard exists to block.

**Decision — ship A.** It fixes the producer (the digest handing the planner an impossible instruction) instead of guarding consumers, reuses the established once-per-replan fetch cadence (docked replans only — missions need a station, and an undocked tick makes no call at all), and fails soft: a failed or empty fetch renders no mission section and never blocks the replan, mirroring how surroundings degrade. The executor's empty-param accept guard stays as the deterministic backstop, and its blocked-reason text was retargeted at the briefing's listing (it used to tell the planner to run get_missions — the same dead instruction through a side door). The transferable lesson: when a prompt and a validator disagree, the model obeys the prompt and the validator wins — the errors land in between, invisible unless you read both sides. Prompt vocabulary and schema vocabulary have to be maintained as ONE contract; here that contract is now guarded from both sides by tests (the digest never names an unplannable action; PlanSchema still rejects query steps).

## 2026-07-13 — Community-harness evaluation #2: squad (bradygaster) — adopt two practices, validate the skeleton

**The problem/context.** The operator asked where our team-making approach diverges from `bradygaster/squad` — an alpha, Copilot-based AI dev-team product (markdown roster/charters committed to git, a 1,076-line coordinator prompt, four built-in agents, a standalone GitHub-issue poller) — and what's worth adapting. Same exercise as the gantry/Zoea interface evaluation (M-19): study the outside design, steal ideas rather than code.

**What the comparison showed.** The two systems converged independently on the same skeleton — a dispatcher that never does domain work, file-mediated agent state versioned in git, author-never-reviews, per-agent model tiering — which is meaningful mutual validation from a stranger solving the same problem. The divergences split cleanly: squad is ahead on *unattended-loop hardening* (its watch poller has tiered error escalation, sentinel-file shutdown, pause windows, a health probe) and *mechanical memory hygiene* (byte-threshold gates that force archival of its shared memory files); we are ahead on *enforcement* (squad's behavioral contract is prompt prose full of ALL-CAPS pleading — including a rule documenting its own past failure — exactly the pattern our forcing-function hierarchy exists to break), *verification* (squad has no eval story, no CI-green definition of done, trusts agent self-report), *review quality* (approve/reject vs our correct-AND-smallest), *cost discipline* (its "eager execution — launch anticipatory agents" doctrine is the opposite of our concurrency cap and would be waste in our cost model), and *operator truthfulness/pushback*.

**Options on the table.** (1) Adopt nothing — defensible, most of squad's surface solves product problems we don't have (casting personas, ceremonies, state backends, marketplaces). (2) Adopt the two cheap gap-fills and file the loop-hardening checklist — the researcher's recommendation. (3) Adopt more aggressively (Fact Checker devil's-advocate agent, skill confidence lifecycle) — rejected: our pushback agreement and invariant-promotion rule already cover both with better triggers; a standing DA agent is busywork.

**Decision: option 2.** Adopted (a) **revision lockout** — a reject-with-defect task is re-dispatched to a fresh implementer, never revised by its authoring context (extends never-self-review from *review* to *revision*; our ephemeral-worker design made this nearly free but it was unwritten); (b) **doc-steward size gates** — numeric byte thresholds that trigger mechanical archival of decisions.md/STATE.md (we had a compaction *protocol* but no compaction *threshold*; this is working-agreement #5 — checked preconditions over remembered ones — applied to doc hygiene). Filed squad's five watch-mode resilience patterns on the durable-scheduler gap (#114) as the checklist for when we build it. Skipped directive capture (our memory/decision-log discipline is strictly richer) and everything else by the value-density test.

## 2026-07-13 — Security frameworks: adopt SSDF + SLSA + OpenSSF Scorecard; reject CSF 2.0, SAMM, Badge, Allstar

**The problem/context.** The operator directed a security-team assessment of NIST CSF 2.0, OWASP SAMM, SLSA, and NIST SSDF (OpenSSF added mid-assessment), choosing one or two with real security value for us and in the real world, then aligning: documented controls and standing oversight of the dev teams. Two independent seats ran it — a framework analyst (outside-in: what each framework is, real-world traction, fit) and a read-only posture auditor (inside-out: our verifiable controls with file:line evidence, mapped to each framework's control families, gaps ranked by real risk).

**Options on the table.** (1) CSF 2.0 — massive institutional traction (insurers, examiners, post-FFIEC-CAT migration), but it's org-risk governance whose adoption currency is evidence for boards and auditors this project doesn't have; both seats independently scored it "awareness, not fixes." (2) SAMM — the only candidate that assesses the TEAM rather than artifacts (steelmanned as a maturity curriculum for a team-building project), but its interviews presuppose human org structures, its benchmark is thin, and it yields scores where we need controls. (3) SLSA — narrow build/supply-chain integrity; we're already near Build L2 by construction; its one forcing gap (provenance + verification) is a real, cheap control. (4) SSDF — the enduring US-procurement practice vocabulary (EO 14028 lineage; survived the 2025-26 EO churn), practice-oriented and proportional, with 800-218A as the only AI-specific text in any candidate. (5) OpenSSF family — Scorecard as continuous mechanical enforcement (CLI works on private repos; badge/score/publish are public-OSS signaling we ignore); Best Practices Badge and Allstar rejected (signaling paperwork; a third-party write-access grant for a one-repo org). A leaner alternative to Scorecard (zizmor + OSV-Scanner) was recorded: Scorecard wins on the transferable operate-the-industry-scorer skill — a tiebreaker, not a security argument.

**The decision.** Adopt three, each at a different rung of the forcing-function ladder: **SSDF** as the audit vocabulary (gate tier — milestone-gate gap analysis with dated deltas against the new `docs/wiki/security-controls.md` register), **SLSA Build L3** as the enforced technical control (provenance in CI + fail-closed verification in the pull deployer, #171), **Scorecard** as continuous enforcement of repo-hygiene invariants (automate tier, #172). Both seats converged on SSDF+SLSA independently — the analyst from traction and design, the auditor from measured gaps — and the OpenSSF addendum complemented rather than changed it. The analyst's pre-mortem (mapping rots as a wiki page; verification left warn-only) is countered structurally: fail-closed from day one, and audits that don't update the register didn't happen. The posture audit's #1 real risk (the dashboard's missing second auth barrier, a defense-in-depth violation independent of any framework) is filed as #173.

## 2026-07-13 — Catalog-gated jettison + market-aware disposal: value decides, not one market's silence (#94)

**The problem/context.** The jettison decision above (2026-07-12) shipped jettison as "the guaranteed disposal escape" with an unconditional briefing line: when nothing reachable buys an item, dump it. That teaching had no value distinction, and the live pilot applied it to exactly the wrong cargo: 28 Palladium Ore — catalog `base_value` 200cr, rare, an input to 5 recipes — held and repeatedly framed as "dead weight / ignore entirely per standing instructions" across plans (events 12237–12424). The operator mandate (issue #94 comment, 2026-07-13) settled it: Palladium Ore is NOT dead weight; it can be sold to players or used in crafting. "No NPC buyer right now" was being read as "worthless," and one briefing rewording away sat ~5600cr of destruction. The invariant: valuable cargo must never be destroyed — and the producer of the bad state was our own digest line teaching jettison unconditionally.

**Options on the table.**
- *(A) Briefing-only correction.* Rewrite the digest line to distinguish worthless from valuable cargo, no code. Cheapest, but the palladium incident is precisely a prose rule failing against a model's accumulated framing ("standing instructions" outvoted a newer operator steer, #186) — a prose-only fix leaves the destroying call one bad plan away. Rejected as the sole fix; kept as the "pick right the first time" layer.
- *(B) Catalog-gated executor guard + corrected briefing [SHIPPED].* Deterministic producer-side check at the jettison seam: refuse any jettison whose catalog `base_value` clears a floor (50cr — common junk ore runs 4–8cr, the incident item 200cr; the constant's receipt lives at `JETTISON_VALUE_FLOOR` in `src/agent/executor.ts`). The refusal is a blocked wake whose reason names the value and the alternatives (hold / re-check markets when docked / `create_sell_order`), alternatives-first so the digest's 200-char clip can't cut them. Fails open on an item the catalog doesn't value — never fabricate a block from missing data (same convention as the mine guard). This is the pattern the harness already trusts: the model chooses, deterministic code makes the catastrophic choice impossible.
- *(C) Register `create_sell_order` now vs. keep deferring it.* Deferred twice before (YAGNI in the auto_list entry; missing pricing signal in the jettison entry). Both deferral reasons have since expired: the live need appeared (valuable cargo with no NPC buyer is exactly the case it exists for, and the guard now actively points the planner at it), and the catalog SSOT (#104) supplies the pricing signal the last entry said to wait for — `base_value` as the deterministic default price. Registered: `spacemolt_market/create_sell_order(item_id, quantity, price_each?)`, price_each optional on our side, filled from the catalog `base_value` by the executor when omitted; with neither a price nor a catalog value the step blocks for an explicit price rather than send an unpriced listing (the game's behavior there is uncaptured). The `orders` batch property is skipped — no captured shape, no guessed layouts.

**The decision.** B + C together, plus the paired improv-mode rule (§4 worthless-vs-valuable, §5 backstop entry) the AGENTS.md convention requires and a parity-test seam so the pairing can't silently rot. The guard is the fix at the producer (the harness's own teaching created the framing; the harness's own executor now bounds its worst consequence), and C is a reversal on the record: what was correctly YAGNI twice became correctly in-scope the moment the need and the pricing signal both existed. The 2026-07-12 jettison entry's guidance is superseded for VALUABLE cargo only — jettison remains the right escape for genuinely worthless junk, which is why the guard has a floor instead of a ban.

## 2026-07-13 — The main seat is an Orchestrator, not a PM: rename, keep accept authority, codify the tech-lead trigger

**The problem/context.** The operator observed that no real-world PM performs merges or consumes code reviews — those belong to team/dev leads, while a PM coordinates, shuffles priorities, reports, and watches timelines. The main-context role here does both sets of duties, so either the role is overloaded or the name is wrong.

**Options on the table.**
- *(A) Split now, human-style.* Stand up a permanent Tech Lead layer and move merges there today. Rejected: waves are currently two PRs wide under the half-pace posture — a standing lead adds a coordination hop and token spend with nothing to shepherd, and moving merge AUTHORITY (not just mechanics) to ephemeral agents was already rejected by council (unattended agents flag merge-ready, never merge).
- *(B) Status quo.* Keep the name "PM" and the duties. Rejected: the name actively misleads — it invites exactly the (correct) challenge the operator raised, and mislabeled roles get mis-reasoned about when charters and scheduler prompts are written from them.
- *(C) Rename + trigger [CHOSEN].* Rename the seat Orchestrator (what it does: coordinate, prioritize, report, interface with the operator, and hold ACCEPT authority — merge-on-reviewed-ADVANCE as a decision consuming a delegated review, not a technical act). Codify the dormant tech-lead layer''s activation as a checked precondition: >2 concurrent lanes in a wave, or #114 scheduler live → batch-shepherding and merge mechanics move to a Tech Lead charter; accept authority never moves. Rationale is L-23 (translate the function, not the form): the human PM/lead split solves a human capacity constraint this system doesn''t have, but the human intuition — the coordinator shouldn''t be doing technical work — is honored by the fact that technical judgment is fully delegated already.

**Also tightened in the same change:** policy-document edits (working agreements, charters, team-structure) now require independent review before merge — the seat had been self-merging its own rule changes under the STATE-freshness exemption, which was written for the handoff file, not for rules. STATE NOW refreshes, generated backlog, and typo-class changes stay exempt. This entry''s own PR is the first exercise of that rule.

## 2026-07-13 — Worktree-isolation gate: a hook denies un-isolated repo-writing dispatches (#192)

**The problem/context.** For the second time, a repo-writing agent was dispatched into the shared checkout: the post-#189 doc-steward, sent without `isolation: "worktree"`, committed its reconciliation onto the branch the orchestrator was editing at that moment. The first occurrence created the written rule; the second occurrence violated it anyway. Per invariant promotion, a rule missed twice stops being prose and becomes a forcing function.

**Options on the table (from issue #192).**
- *(1) A dispatch-instructions line in each recurring-role charter.* Rejected in the issue itself: charters are inlined by the same seat that forgot — it is the same prose, one file over.
- *(2) A `PreToolUse` hook on the Agent tool [CHOSEN].* Deterministic, and fires at the exact moment of the mistake: when the dispatch's `subagent_type` cannot be proven read-only and no `isolation` is set, the hook denies the call with a message saying how to fix it or consciously override (`ISOLATION-OVERRIDE: <reason>` in the prompt — the reason text is required, a bare token does not pass). "Repo-writing" is not a hand-kept list: the hook derives it from the `tools:` frontmatter of `.claude/agents/<type>.md` (exempt only when every granted tool is provably read-only; no `tools:` field means all tools), hardcodes only the built-ins that have no definition file (Explore/Plan read-only; general-purpose, fork, and an omitted type — the tool's default — all-tools), and treats unknown types as requiring isolation. The first cut of this hook DID use a hand-maintained constant, and its independent review (PR #201) found the list had already drifted: `docker-expert`, a role defined in the same repo with Write/Edit/Bash in its grant, sailed through — the same "rule that lives only in something a human has to remember" failure this whole gate exists to kill. Deriving from the frontmatter fixes the producer. The trade: charter roles with no `tools:` restriction (task-reviewer, soc-monitor, …) now need isolation even when their charter says read-only — the gate trusts the grant, not the prose, and a worktree that nothing writes to is auto-cleaned for free.
- *(3) Fold into #114's spawner (worktrees by construction).* Not rejected — deferred: the spawner doesn't exist yet, and manual dispatches from the main seat would still bypass it. The hook covers today's path and stays useful as a backstop after #114 lands.

**Two deliberate deviations, with receipts.** The hook DENIES rather than warns — the existing hooks are WARN-only, but this incident is the documented case where warn-prose re-breaks. And it is written in bun/TypeScript rather than the sh of the other hooks — the decision needs real JSON parsing, because a raw-text grep (the sh convention) would pass any dispatch whose prompt merely *mentions* `isolation: "worktree"`. Fail-open throughout: any hook error logs and allows, so a broken gate can never brick dispatching. Details and tests: `.claude/guardrails.md`, `test/agent-worktree-gate.test.ts`.

## 2026-07-14 — Dashboard second auth barrier: gate everything, receipt the .env deviation, ship disabled-until-provisioned (#173)

**The problem/context.** The posture audit's #1 real risk: the dashboard trusted every caller that reached its port — the SSO provider at the proxy was the ONLY barrier, and POST /instruct is a write path into the planner prompt. One proxy misconfig or any front-network container = unauthenticated planner injection. Fix: an in-process shared-secret header (`X-Dashboard-Token`, injected by reverse-proxy middleware) checked before anything else. Three choices inside that fix were consequential enough to log.

**Choice 1 — what the gate covers.** The issue sketched "mutating routes at minimum." Shipped instead: the check is the first statement in `fetch()`, before routing — HTML, API reads, POST /instruct, WS upgrade, and every FUTURE route covered by default, at the same line count. A route allowlist would re-open on every route added; gate-before-routing can't drift.

**Choice 2 — where the secret lives.** (a) `secrets/`-only with `_FILE` works for the app side but reverse-proxy compose labels have no `_FILE` mechanism — the middleware value must interpolate from the environment. (b) Reverse-proxy file-provider middleware (like the SSO provider's own dynamic config) would move the definition into the deployment's shared dynamic config — but that lives in a different repo, and the file provider has no secret-file interpolation either; the same rendering problem, relocated to infrastructure this repo doesn't own. Rejected. (c) SHIPPED: `DASHBOARD_TOKEN` in the stack `.env` (0600) for the label, the same value bind-mounted as a real secret file for the app — a RECEIPTED deviation from the ".env holds non-sensitive config only" rule: exposure is equivalent to what docker-inspect access on the host already grants, documented in .env.example, both compose files, and the controls-register delta (security.md "Accepted tradeoffs" precedent).

**Choice 3 — startup behavior.** Always-refuse-without-token would brick the auto-deploy: the image lands ~1min after merge, before any host provisioning (the deployer pulls images only, never compose). Shipped: knob absent → disabled with exactly one loud warning (deploy-safe); knob set but file missing or EMPTY → refuse startup (fail-closed where misconfiguration is provable). The `${DASHBOARD_TOKEN:?}` compose interpolation hard-fails the manual apply if `.env` wasn''t provisioned — no half-configured state is reachable. Named residual (accepted per deployment proportionality, security.md skip tier): no automated alert if a future compose regen silently drops the knob; detection is the deploy checklist''s log check.

Source: issue #173; PR #202 (adversarial review verdict with ablation evidence on the PR); docs/wiki/security-controls.md delta.

## 2026-07-14 — Instruction supersession: a newer operator steer must beat an older conflicting one (#186)

**The problem/context.** The palladium incident's root cause, seen live: `Agent.instruct()` pushed every operator steer onto an unbounded goals list that the digest rendered oldest-first with no recency rule. The operator's newer "sell palladium if a buyer is detected" steer (instruction wake, event 12418) produced one compliant plan — and the very next plan (12424) reverted to the OLDER accumulated "treat palladium as dead weight" framing, calling it "standing instructions." The invariant: a newer operator instruction beats an older conflicting one. The catalog-gated jettison guard (entry above) bounds the worst consequence; this fixes the precedence defect itself.

**Options on the table.**
- *(A) Newest-first briefing + explicit supersede rule [SHIPPED].* The digest renders goals newest-first (copy-then-reverse — `ctx.goals` is the agent's live array, never mutated) with one rule line: when two conflict, the newer supersedes the older.
- *(B) Bounded history [SHIPPED with A].* `MAX_GOALS = 5`, evicting oldest, enforced at BOTH producers: the `replan()` push and the constructor's reload of persisted goals (goals persist with the plan via `store.savePlan`, so a stored list predating the cap can exceed it — trimmed on load, never a crash; the persisted-state-tolerance convention, tested against a real Store round-trip). Receipt for 5: steers are rare and operator-typed; five holds a session's worth of live intent while guaranteeing stale steers age out. Behavior change accepted: an instruction more than 5 steers old vanishes from the briefing instead of persisting until restart.
- *(C) Revocation verb / TTL machinery / conflict-detection.* Rejected as gold-plating — the issue records them as alternatives; nothing in implementation showed A+B insufficient.

## 2026-07-14 — Image provenance: key-based cosign signing, verified fail-closed at the deployer (#171)

**The problem/context.** Every image the host runs is digest-pinned, which proves *which* bytes are running — but nothing proves *how they were built*. Anyone holding a GHCR write credential (a leaked PAT, a compromised CI token) could push an image to our package and the auto-deployer would run it within a minute. The security-team adoption (#174 entry above) named this the one new enforced technical control: provenance generated in CI, verified fail-closed by the pull deployer. The operator's calibration on the issue reframed the target — the control, not the SLSA level number — with a threat-model note that fits this project unusually well: our build steps are authored by LLM agents with a known injection surface (game text), so "defend the signature from your own developers" is accurate, not enterprise paranoia. One hard fact set the design space: GitHub's native artifact attestations require an Enterprise Cloud plan on private repositories, and this is a personal-plan private repo — verified during implementation, so the issue's first-choice path was closed before it started.

**Options on the table.**
- *(1) GitHub-native attestations* (`actions/attest-build-provenance` in CI, `gh attestation verify` in the deployer). The genuine L3 path: provenance signed by GitHub's own infrastructure, isolated from our build steps, no key to manage. Rejected by entitlement, not on merit — on a private personal-plan repo the attestation API refuses. It would also have put `gh` plus a GitHub API token on the NAS host (apt-locked, so a manual static install, plus a new credential where cosign needs none).
- *(2) Cosign keyless* (Sigstore public-good: the workflow's OIDC identity gets a short-lived certificate, no long-lived key). Stronger than a key in principle — nothing to exfiltrate, the signature is bound to the exact workflow identity. Rejected for a private repo: every signature lands in the public Rekor transparency log, publishing the image path and build metadata of a repo we deliberately keep private; and host verification needs the Fulcio/Rekor trust roots kept fresh, more moving parts on an apt-locked host.
- *(3) Key-based cosign [SHIPPED].* A keypair generated once: the private half exists only as GitHub Actions secrets (encrypted, passphrase-protected), the public half is committed in-repo and copied to the host. CI signs a SLSA v1 provenance predicate against the exact pushed digest with `--tlog-upload=false` (nothing to the public log — the key is the trust root); the deployer verifies with the pubkey and `--private-infrastructure`. The issue itself named this the fallback: same design, slightly weaker trust. The honest label is L2 "L3-shaped": the predicate is self-attested by the workflow rather than an isolated builder, and a long-lived key can in principle be stolen where a keyless identity can't.
- *(4) Sign but don't verify (or skip both).* Rejected without debate: the adoption pre-mortem is literally "verification left warn-only after one false positive" — a signature nobody checks is decoration, and a warn-only check is the same thing with extra steps.

**The decision.** Option 3, with the verification gate fail-closed from day one: a verification failure keeps the running image, refuses the deploy, and alerts urgent via the deployer's existing alerting path — exactly the shape of the health-gate failure it sits beside. Three inner choices worth recording. *Ordering:* CI pushes the sha7 tag, attests its digest, and only then moves `latest-main` — the deployer watches `latest-main`, so the gate can never false-trip on a digest whose attestation hasn't landed yet. *Rollback is exempt from the attestation check only:* the auto-rollback path deploys the previous pin without re-running the cosign check (its attestation was verified when it first deployed, and a verification outage must not convert fail-closed into fail-broken — unable to roll back is a worse failure than the one being defended against), but the deployer persists the verified digest alongside the pin and rollback still rechecks the pulled tag against it: a pure local comparison with no registry or attestation dependency, so the tag-repoint window stays closed on rollback too. *Version pinned across the pair:* cosign v2.6.3 on both sides — `--tlog-upload` is deprecated in v3 (as of v3.0.3; still functional, removal deferred to v4), and v3's default push toward signing-configs aimed at the public Sigstore services is an offline-untestable change — pin v2 until the v3 path is deliberately evaluated. The v2 line still gets releases (2026-04). The deployer script lives in the private infrastructure repo, so the host half ships as a complete runbook there the PM applies after the first signed image publishes; until then images are signed but unverified. The issue's folded-in dependency-change-as-task CI gate is NOT part of this change — split out to #210, which carries that remainder.

## 2026-07-14 — The planner kept aiming at things it could not touch: target locality, checked at run time (#176)

**The problem/context.** The 6-hour strategy review, mining the durable event store, surfaced two of the pilot's worst live failures. `scan` was a 100%-broken capability — all 16 lifetime attempts rejected, every recent one `invalid_target: Target 'commerce_fields' not found at your current location`. And roughly 30 travel attempts in 72 hours came back with variations of "Gold Run Mineral Fields is in the Gold Run system (gold_run), but you are in market_prime." Every one of those is a wasted tick plus a wasted LLM call to replan around a self-inflicted error. Both look like two bugs. They are one: the pilot kept naming a target it could not act on from where it actually was.

Two things make that happen, and they are worth separating because the second one is a genuinely subtle trap.

First, `scan`. A star system is described by `get_system`, which returns the system's points of interest — the belt, the star, the station. Those POIs were the only ids the planner had ever been shown. But `scan` does not look at places; it looks at *things at your position* — another ship, a wreck, a drone — and the game lists those under a different query, `get_nearby`, which nothing in the harness had ever called. The planner reached into the only id list it had, which was the wrong one, and did so 16 times out of 16. The game's own error even names the right query. Nobody was reading it.

Second, `travel`. The harness already checks a plan's POI ids against the system the pilot is in — a plan naming an unknown POI is rejected before it costs a game call. That check runs when the plan is *written*. The trap is that a plan is not an instant; it is a sequence executed over many ten-second ticks, and some of its steps *move the ship*. The pilot writes a perfectly sensible round trip while sitting at a mining belt: fly to the market, dock, sell, come back to the belt. Every id in it is valid at the moment it is written. By the time the last step runs, the ship is in another star system, where that belt does not exist. The plan was correct when admitted and stale when executed. That is the whole bug: plan-time validity is not execution-time validity.

**Options on the table.**
- *(A) Read the target system out of the game's error text and auto-convert the doomed `travel` into a jump-then-travel.* Tempting, since the error helpfully names the destination system. Rejected: it is a patch at the crash site, applied *after* the wasted call — the exact waste being eliminated. It also teaches the harness to parse English error prose, which changes without warning.
- *(B) Teach the plan-admission check to simulate the ship's position across the plan's steps.* Correct in principle, and it fixes the producer where the bad plan is born. Rejected as the primary fix: it means modelling the positional effect of every action for a check that a single free query answers exactly at the moment it matters. Complexity with no receipt.
- *(C) Check locality at execution time, in the executor, one step before the call goes out [CHOSEN].* At that instant the ship's real position is a free, unlimited query away. If a `travel` names a POI that is not in the current system, block it before the wire and say what to do instead ("plan travel_to{system_id} first"). If a `scan` names a POI id at all, block it and point at the Nearby list. This is the same seam and the same shape as the guards already in place for mining without a laser, undocking while undocked, and jettisoning valuable cargo.
- *(D) For scan, simply forbid it in the briefing.* It would stop the failures — and permanently kill a capability that feeds one of the pilot's progress dimensions. Rejected: fixing the id source is barely more work than banning the action.

**The decision and why.** (C), plus the missing id source. The executor now blocks both classes before the call, and each block names the alternative rather than failing silently — a blocked step wakes the planner with that text, so the correction reaches the model on the very next plan. The harness also fetches `get_nearby` on every replan and renders it in the briefing directly beneath the POI list, each list labelled with what it is *for*: these are places you travel to; these are entities you scan. The raw listing text is passed through unparsed — that response has never been captured live, and this project has been bitten before by parsing a shape it guessed at. Both guards fail open: if the map query is unavailable, no block is fabricated from missing data.

The deeper lesson, and the reason this one is worth the ink: a validation that runs at plan time silently assumes the world holds still until the plan finishes. It does not. Anything a plan *changes* invalidates the checks that ran before it — so a precondition about *where you are* has to be checked *when you act*, not when you decide. That principle is not about spaceships.

## 2026-07-14 — The pilot could not spend its money: registering the ship tool (#219)

**The problem/context.** The miner had 17,306 credits in the bank and had never bought anything but fuel. Not once, in its entire recorded life. Two issues had been opened about it (#107, #216) proposing better goals and sharper briefings, on the theory that the pilot was being told the wrong things.

It was not. Look at what the harness could actually *do*. Every game action the pilot is allowed to take is declared in one file, `src/registry/actions.ts` — a deliberate choke point, so there is exactly one place to look for "what can this pilot do?". Nothing in that file could browse a shipyard, buy a ship, or fit a module. The game supports all three. We had simply never wired them. No amount of goal-setting fixes an action that does not exist: telling a pilot to buy a better mining laser when it has no way to buy or fit one is telling it to do something impossible, and it will (correctly) go back to mining ore.

This is a repeat of a shape we have hit twice already — mission actions unregistered while we wondered why the pilot never ran missions (#124), and remote-POI targeting broken while we wondered why every scan failed (#176). The game could do it; we never plugged it in.

**Options on the table.**
- *(A) Register the actions and stop there.* The minimum that makes the purchase *possible*. Rejected as insufficient on its own, and we have the receipt: `browse_ships` is a free query, and by design the planner can only ever plan *mutations* — so it structurally cannot call it. An action the planner is never shown the output of is an action it never uses. This is exactly what happened with missions (#147), where the briefing told the model to call a query it was incapable of planning, and eleven planner errors later nothing had changed.
- *(B) Register them, and have the harness fetch the shipyard listing itself and show it in the briefing [CHOSEN, with (C)].* The established pattern: the harness runs the free query and hands the planner the results — the listing ids, the prices — as data. The planner then only has to plan the purchase.
- *(C) Add a deterministic pre-flight check on fitting [CHOSEN].* A module has to fit the hull's CPU budget, its power budget, and a free slot of the right type. Those are hard caps. The game offers no dry-run for a fit, so a bad one costs a full ten-second tick and a wasted planning call. Our own status query already carries the ship's grid, and one free catalog lookup gives the module's cost — so the check is a comparison of numbers we can get for nothing, made one step before the call goes out.
- *(D) Guard the skill requirements too.* Rejected. Modules can be gated on a skill level, and the game's schema lists a `required_skills` field whose values are integers — a level, plainly. (An earlier draft of this entry claimed the field had "no value shape at all" and marked that verified; it was wrong, and the review caught it. The schema does specify the value type. What it does not specify is the *keys*: which skill names can appear there, and whether they are spelled the same way `get_skills` spells them.) That is enough to make the gate unsafe to build, but it is not the main reason. The main reason is the one below: a client-side gate that gets it wrong *blocks a purchase the game would have allowed*, and unlike the CPU/power check there is no safe floor to fall back on. The game already enforces the skill gate itself, for free, in its own rejection message. So we leave it to the game, and write down the capture that would let us reconsider.

**The decision and why.** All three of (A), (B) and (C), because none of them works alone: registration makes the action *possible*, the briefing makes it *reachable*, and the guard makes it *cheap to get wrong*. The briefing now shows the pilot its own ship — CPU used, power used, free headroom, which slots are taken — right next to the credit balance, because "can I afford the next laser?" was never a question about money. It is a question about CPU and power, and the planner had never once been shown either.

**A guard that can be wrong in the wrong direction is worse than no guard (the review's correction).** The first version of the fitting check compared the catalog's stated CPU and power cost of a module against the ship's free grid, and hard-blocked anything that did not fit. An independent review found the hole. The game has an Engineering skill that makes every module cheaper to run — one percent off its CPU and power for each level — and the game's own documentation says the numbers you see when you inspect your *ship* already have that discount baked in. It says nothing about whether the numbers in the *catalog* do. Nobody has ever captured the answer. So the guard was resting on a coin-flip: if the catalog quotes the full undiscounted price, then a trained engineer's module that fits perfectly well in real life would be refused by our own code, before the game ever heard about it — and the pilot with 17,306 credits still could not upgrade. We would have shipped, as the guard, the exact failure the guard exists to prevent.

The fix is not to guess better; it is to make the guess impossible to get wrong. The discount can only make a module *cheaper*, never more expensive. So we compute the floor: the smallest the real cost could possibly be, given the pilot's actual Engineering level (a free query the harness already knows how to make). If even that floor does not fit, the module cannot fit under either reading of the catalog, and blocking is safe. If it might fit, we let the call go to the game and let the game answer — its rejection is authoritative, it explains itself, and the harness already routes a rejection back to the planner as a wake-up. The guard keeps its whole point, which is catching the hopeless fit for free, and loses its ability to lie. A capture of one fitted module's numbers — its cost as the ship reports it, next to its cost as the catalog reports it, on a pilot with any Engineering at all — settles the question for good and would let the check tighten. Until then, the safe half is the half we ship.

One wrinkle worth recording for whoever wires the next tool. Every other endpoint in this game lives at `/<tool>/<action>`; the catalog lives at a bare `/<tool>` with no action at all. Rather than invent a general "custom path" mechanism for a single exception, the registry gives that one entry an empty action name and the transport drops the trailing slash. One ternary, one comment, no new machinery — and the offline fake server routes it the same way, so the only transport shape we could not otherwise test is now covered by a test.

The transferable lesson (harness engineering): when an agent will not do the thing you keep asking it to do, check whether it *can* before you rewrite the prompt. A capability gap and a motivation gap look identical from the outside — the agent just keeps doing something else — but they have nothing in common underneath, and only one of them is fixed by better instructions. The registry file is the fastest place to tell them apart.

## 2026-07-14 — Plan context replay: fidelity via a single builder with enumerated schema, not per-field clipping

**Problem.** A test for plan_context replay fidelity (how exactly the game state is copied into the event that powers an eventual improv mode) needed to assert that each field stores what the planner saw, no more no less. Writing the test exposed a fixture trap: uniform filler (`"x".repeat(n)` everywhere) made the assertion vacuously true — a 200-char substring of all-x is identical to any clip of the same string, so the test passed even when the clip was wrong. **Context.** The harness clips, compresses, and structures context as it moves through the pipeline. **Decision — (B).** The builder's guarantee is stronger: every input the built output depends on is captured in the schema and verified at compile time. The integrator verifies the seam, so landed code still passes the same gates as sequential work would have. This shape is now the pattern for same-file-append parallelism and is documented in docs/wiki/engineering-lessons.md as the operational lesson.

## 2026-07-19 — Deterministic 5xx step retry, movement actions only (#431, PR #442)

**Context.** On 2026-07-19 the game answered travel with HTTP 503, and each 503 became a full planner call: the step blocked, the planner woke, and the fresh plan's first step was byte-identical to the one that had just failed (about 4 minutes and one LLM call per cycle). The same class hit travel and get_status on 2026-07-13; a failure class seen twice becomes a deterministic constraint (invariant promotion). Replanning cannot fix a 503. The remedy is to resubmit the same step after a pause, which code does for free. The transport (http.ts) already retries a 5xx in-call for about 7 seconds, so the open question was where a second, slower retry tier should sit and how widely it should apply.

**Options.**
- *(A) Retry every plan step on a 5xx [REJECTED].* Plan steps are all mutations; an ambiguous 5xx can land after a server-side commit, so a blind resubmit of sell/buy/mine reopens the #137 at-least-once double-spend class the buy path already fenced off, tripling transport tries to about 9 with roughly 27-second gaps for an async commit to land first.
- *(B) A bigger retry budget inside http.ts [REJECTED].* Blocks the tick loop for minutes inside one call and hides the whole episode from telemetry.
- *(C) Movement-gated executor retry [CHOSEN].* travel/jump/travel_to converge on re-issue (moving toward where the ship already is costs nothing), and every live #431 incident was movement or status traffic.

**Decision.** (C). The transport classifies (isTransientServerFailure, defined once at the producer; circuit_open included so a mid-outage open breaker cannot masquerade as a terminal block), the executor surfaces a server_retry outcome, and the agent loop consumes it for movement steps only: 3 total submissions, 2 ticks between them, a step_retry_5xx event per retry, then the ordinary blocked wake when the cap is spent. Non-movement mutations block immediately, exactly as before; one wasted planner call beats a double-spend. Counters live in memory only, so persisted state is untouched. Paired improv-mode note sits in the spec's section 5 as a deterministic backstop that stays deterministic. Depth: issue #431, PR #442.

## 2026-07-19 — Buy-side escrow reclaim: register cancel_order, defer a credit floor (#322)

**Context.** Posting a standing bid with `create_buy_order` moves credits out of the wallet and into escrow the instant the order is placed (upstream/docs/markets.md:31). If that bid never fills, the credits sit locked. Issue #322 (from the #319 review) named two gaps: no way for the harness to reclaim a stranded bid, and no wallet bound stopping the pilot from escrowing more than it should. As this entry was written (2026-07-19) the pilot held roughly 8,130cr in two open bids, so the reclaim path is live-relevant, not hypothetical.

**Options.**
- *(A) Register `cancel_order` and rely on the game's wallet ceiling [CHOSEN].* Gives a deterministic reclaim: `view_orders` lists the open orders, `cancel_order(order_id)` returns a buy order's remaining credits to the wallet and a sell order's remaining items to storage (spec description, openapi-v2.json:99293). This is the game's own remedy, not a new harness primitive.
- *(B) Also add a harness-side credit floor before buy orders [DEFERRED].* A reserve guard that refuses to escrow past some balance. The #322 reviewer assessed it as a new primitive needing a receipt: `create_buy_order` and `create_sell_order` are equally unbounded and the game already enforces the wallet ceiling, so a floor extends no accepted posture and defends against a failure not yet observed live.

**Decision.** (A). `cancel_order` was registered by the 2026-07-19 capability audit (PR #422, actions.ts:158; commands.md:337 flipped to ✅), which closes the reclaim gap independently of this decision. Shape verified against the vendored OpenAPI (openapi-v2.json:99291-99704, operationId `spacemolt_market_cancel_order`): properties `[order_id, order_ids]`, empty required set, `x-is-mutation:true`. The credit floor stays deferred because it is a new primitive with no live receipt, and the simpler cause-site fix (reclaim rather than prevent) already exists. Revisit only when live evidence shows the game's own wallet ceiling insufficient. At that point the floor earns a fuel-reserve-style spec and a decision entry of its own.

## 2026-07-10 — Thrash damper: 3 consecutive same-reason blocks arm a cooldown (F-3; migrated from a code comment, issue #121)

**Context.** The maiden flight (the SM-1 through SM-5 campaign covered above) showed every blocked step immediately woke the planner for a fresh plan: about 60 replans an hour against a 4-10/hour design target, each one a paid LLM call for a failure that often repeats the identical way. The fix needed a threshold high enough to tolerate a real two-step recovery (blocked, replan, one exploratory step that also blocks, replan succeeds) but low enough to catch a real loop before it burned the plan budget. This entry did not exist when the fix shipped; it captures the ground truth a code comment in `src/agent/agent.ts` (`BLOCKED_THRASH_THRESHOLD`) was carrying alone.

**Options.** Only one mechanism was built and flown; no alternative threshold or approach is on record.
- *(A) 3 consecutive blocked wakes sharing the same detail string arm a plan-call cooldown [SHIPPED].* Caps the burn at roughly a fifth of the observed thrash rate before deferring to the heartbeat cadence, while still tolerating a legitimate two-step recovery.

**Decision.** Ship the 3-strike threshold. It was revised twice in the field, both already told in full above: SM-4 found a no-op `travel_to` (same-position short-circuit) advanced the plan cursor before the identical block recurred, so the original cursor-based zero-progress check never engaged — the trigger became block-REASON repetition instead, which subsumes the zero-progress case without a cursor to spoof. SM-9 then found the damper's blind twin. A plan reporting `plan_done` every cycle with zero verified progress (the phantom-sell episode in the correction entry above -- 17 sell steps over ~20min, cargo unchanged item-for-item: trade_crystal 2, vanadium_ore 31, palladium_ore 20, carbon_ore 47, before and after; credits flat at 304) never tripped the "blocked" branch at all, so the same counter, threshold, backoff, and event were extended to also arm on 3 consecutive `plan_done` wakes completing the identical goal text — keyed by a compound `"reason:detail"` string so the two thrash kinds never cross-contaminate one another's streak.

## 2026-07-13 — Thrash key normalizes to outcome CLASS, not literal detail (issue #146; migrated from a code comment, issue #121)

**Context.** The block-reason-repetition thrash damper (see the F-3 entry above) keys on an exact detail string. A no-buyers sell loop escaped it for 40-plus minutes: every replan tried to sell a DIFFERENT ore at a market with no buyer, so each attempt's blocked-reason text named a different item and no three-in-a-row streak ever built, even though every attempt failed the identical way.

**Options.**
- *(A) Normalize the thrash key to the outcome CLASS before comparing [SHIPPED].* At the single place the key is built (`src/agent/agent.ts`'s wake branch), detect a no-buyers sell block regardless of which item it named and collapse it to one shared class token; every other block reason still keys on its own literal detail string.
- *(B) Enumerate every no-buyers item as its own reason [REJECTED, never seriously considered].* Needs an ever-growing item-id list and still evades any blocked-reason class not yet enumerated — the same losing game the SM-10/SM-11 transient-block fix already ruled out for matching block text.

**Decision.** Ship (A), a producer-side fix: the damper mechanism itself stays untouched, only the key it receives changes, and the class token flows into the emitted `plan_thrash_backoff` event so the dashboard shows what actually armed the gate. The lesson repeats SM-9's. A thrash detector keyed on a literal string is only as good as its enumeration of what "the same failure" looks like. The fix normalizes toward the failure's CLASS instead of chasing every surface variation the game can express it in.

## 2026-07-20 — Store-API transport for the scheduler container: TLS carve-out at the reverse proxy over a published LAN port (#398 A1 successor)

**Context.** The strategy reviewer on the scheduler container needs the harness store API on the NAS host.
