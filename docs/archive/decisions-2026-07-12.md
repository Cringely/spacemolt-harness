## 2026-07-12 — Near-real-time auto-deploy: a host-side pull, not a CI push

**Problem.** In active development the manual "SSH in and redeploy" step was too slow. We wanted a
successful CI build to reach production within about a minute, without weakening a host that sits
on a private LAN behind a reverse tunnel, the reverse proxy, and the SSO provider. The operator's directive:
near-real-time, and a green pipeline should deploy on its own.

**Context that framed the answer.** The target environment already standardizes on *pull* deployment — a
weekly cron pulls images; an image-watcher only *notifies*, it doesn't auto-pull. There is no
inbound path from GitHub to the host and no credential of the host's lives at GitHub. That posture
is the thing worth protecting, so the real question was "how fast can a pull go" rather than
"should CI push."

**Options considered.**
- *Self-hosted GitHub Actions runner on the target environment.* A deploy job runs inside the LAN — seconds
  of latency. Rejected: a runner with the Docker socket is host-level remote code execution, and
  anyone who edits a workflow or lands code on `main` would get arbitrary execution inside the LAN.
  It inverts the target environment's "no CI foothold on prod" posture for a hobby pilot. Blast radius: the
  whole host.
- *Webhook via the reverse tunnel.* CI calls an endpoint on the host that triggers a deploy.
  Rejected: it opens a public, internet-reachable endpoint whose only job is to trigger a
  root-equivalent action, and it needs an SSO bypass (machine-to-machine can't do
  interactive SSO). Most new attack surface of any option.
- *GitHub Actions SSH to the host over a mesh VPN.* Rejected: it puts both a mesh-VPN credential and
  an SSH key into GitHub secrets, so a compromised run or tampered workflow exfiltrates host
  access. Inverts the no-creds-at-GitHub model.
- *Host-side pull on a fast timer.* The host checks the registry for a new `latest-main` digest
  every 60 seconds and deploys on change, health-gates, and auto-rolls-back. Latency: about a
  minute. Keeps the property that nothing at GitHub can reach the host.

**Decision — the fast host-side pull.** It is the only option that delivers near-real-time
(≈1-minute) deploys while keeping "nothing at GitHub touches the host," and it reuses the pull
pattern the operator already maintains rather than inventing a runner/webhook/tunnel. CI's job
stays "publish a *tested* image" (the test suite runs inside the Docker build, so broken code never
publishes); the host decides when to pull. The worst a poisoned image or a bad merge can do is get
caught by that in-image test gate, then bounded by a health-check and automatic rollback to the
last-good pinned sha. That is the right blast radius for a self-hosted pilot; the three push options
all traded that away for latency the pilot doesn't need. Implemented as `spacemolt-redeploy.sh`
(cron every 60s) in the private infrastructure repo; pins the immutable `<sha7>` for reproducible rollback.
Tradeoff accepted: a ~1-minute lag versus a push model's seconds, and a shallow health-check that
catches boot/crash failures but not logic bugs (the in-image test gate is the real correctness
guard). If literal-seconds is ever needed, the self-hosted-runner push is the fallback, with its
documented host-RCE surface.

## 2026-07-12 — Selling where there are no buyers: market visibility

**Problem.** A station's market is not a universal buyer. The live pilot, docked and holding cargo, planned a `sell` at a market that had no buy order for the item it held. The sell came back with no effect. This is not a runaway or a loop — the sell effect-verification we added for SM-9 (compare the item's cargo quantity before and after; a non-error envelope alone is not trusted) catches the no-effect sell after the fact, marks the step blocked, and the pilot replans. So the harm is bounded: one wasted action, not a thrash. But it is still a doomed action the harness sent because the planner had been told something false. The digest's cargo line read "sellable at any station market" and its runbook line said selling docked cargo "is almost always the correct next step." Both are wrong: a station buys only certain items. This is the same class as the station-awareness dock fix (PR #72) — the pilot acting on a precondition the digest misstated — applied to sell.

**Options.**
- *Full deterministic surfacing (the station-awareness pattern, done properly).* Query `view_market` on each dock, parse which of the items the pilot is holding this station will actually buy, render a "buyable here" list in the digest, and key the sell precondition to it — so a sell the market can't honor is never planned, exactly as `[station]` markers stopped doomed docks. This is the strongest fix and the intended end state. It cannot ship now: it needs the `view_market` **response** shape, and we have not captured it. Our slim OpenAPI fixture records request parameters only (it never recorded response bodies), so building the parser today means guessing field names — the precise bug class this project forbids (a guessed-shape parser is how the `find_route` and chat-channel bugs happened). Deferred to a follow-up: capture the real response from the live pilot, then parse against a real fixture.
- *Use `sell`'s `auto_list` parameter.* The game's `sell` can take `auto_list:true`, which posts a standing sell listing instead of failing when there is no immediate buyer. Rejected as the primary fix: it commits the pilot's cargo to an order that may never fill, changing the economic behavior silently, and it is the opposite of what the operator asked for — they want to *see* which stations buy what, not auto-list into the void. Noted as a future economic option (a deliberate "list it and move on" play), not a fix for the visibility gap.
- *Register `view_market` + correct the false briefing + lean on the existing sell effect-verification backstop.* Add `view_market` to the registry as a query so the planner can call it when docked and read this station's prices. Reword the two false digest lines to the truth: a station buys only certain items, a sell can fail if this market has no buyer for what you hold, check `view_market` before selling, and if a sell is blocked try a different market rather than repeating. The SM-9 effect-verification stays the deterministic backstop that catches any no-effect sell that still slips through.

**Decision — register the query and correct the briefing now; defer the full surfacing.** The third option is the honest, certain one: it ships zero unknowns. Registering `view_market` (request shape verified against the OpenAPI fixture — params `category`, `company_store`, `item_id`, `since`, none required, matching exactly) gives the planner a real tool to check a market before selling. Correcting the digest removes a false statement the pilot was acting on; a planner told the truth ("this market may not buy your item; check first") makes better decisions than one told a convenient lie ("sellable anywhere"). We deliberately do **not** render a "buyable here" list or parse any `view_market` response yet, because we have not captured that response and guessing its fields is the exact failure we refuse to repeat. The full deterministic surfacing — query on dock, parse buyable items, gate the sell precondition — is the intended follow-up, blocked only on capturing the real response shape from the live pilot. Capture first, then parse against the real fixture; do not guess. Until then the SM-9 sell effect-verification is the deterministic floor: a doomed sell wastes one action and is caught, never loops.

## 2026-07-12 — Escaping unsellable cargo: auto_list

> [superseded for no-demand cargo — the auto_list-frees-the-hold premise was falsified live; see the next entry, 2026-07-12 "Disposing of no-demand cargo: jettison".]

**Problem.** The market-visibility fix above assumed a doomed sell wastes one action and the pilot moves on. A live incident showed a worse failure: the miner mined 7x Palladium Ore that *no reachable NPC exchange buys at all* — not "this station won't, try another," but "nothing here will." Its only registered sell path was the plain NPC-exchange `sell`, which returns "Sold 0 ... 7 unsold (no buyers)." With no listing or jettison action registered, the cheap-tier (haiku) planner looped 10+ times bouncing between Market Prime Exchange and Cargo Lanes Freight Depot — burning fuel while credits, ore, and cargo all stayed flat. The SM-9 effect-verification correctly marked each sell blocked, but "blocked, replan" just sent the pilot to the next exchange, which was equally doomed. The pilot was structurally trapped by our registry: we had given it no tool that escapes an item nothing buys.

**Invariant violated.** After a sell returns no buyers, the pilot's hold should be able to clear so it can move on — but the registry offered only a mechanism (NPC `sell`) that cannot clear cargo nothing buys. The bad state (a full hold of unsellable ore) is produced by mining an item with no NPC demand; it is consumed by every subsequent sell/travel plan that assumes the hold can be emptied at *some* exchange. The game's intended producer-side escape is `sell` with `auto_list=true`: it lists the unsold quantity on the *player* exchange (the goods are escrowed from cargo to the standing order) instead of failing, so the hold clears and the pilot is free. VERIFIED against `test/fixtures/openapi-slim.json`: `/spacemolt/sell` properties are `[auto_list, id, quantity]`, required `[id, quantity]`.

**Options.**
- *(A) Planner-briefed auto_list [SHIPPED].* Add `auto_list: z.boolean().optional()` to the registered `sell` params (keeping `.strict()` and the required set unchanged, so the registry-conformance test still passes), and add one digest briefing line: when a sell returns no buyers, retry the same sell with `auto_list=true` to list the goods on the player exchange instead of hunting other NPC exchanges. Smallest change, uses the game's own intended mechanism, and leaves the choice with the planner/operator — auto-listing escrows cargo, which has economic consequences (the order may sit unfilled), so a human-legible, planner-driven decision is the conservative default.
- *(B) Deterministic executor auto-retry.* On any "no buyers" sell result, the executor automatically re-issues the same sell once with `auto_list=true`. MORE reliable — the haiku planner just demonstrated it loops 10+ times rather than reasoning its way out, so a deterministic backstop would guarantee the escape regardless of planner quality. Rejected *for now* on two grounds. First, "prove the premise before replacing a mechanism": we have not yet observed whether the briefing alone (option A) fixes the loop; shipping the deterministic retry now would replace the planner's judgement before proving the planner's judgement is insufficient *with the briefing present* (it looped *without* any briefing or tool — a different premise). Second, auto-listing escrows the pilot's cargo into a standing order without the planner or operator choosing to; escrow is an economic commitment, and a silent auto-retry makes it on the pilot's behalf every time an NPC buyer is missing, including transient cases. Deferred, with an explicit promotion condition (below).
- *(C) Register `create_sell_order` / `jettison`.* Give the planner dedicated actions to post a player-market order or dump cargo. Rejected as larger surface than the problem needs: `sell` + `auto_list` already covers "list what no NPC will buy," and jettison (destroying cargo) is a strictly worse escape than listing it for sale. YAGNI — no live need has appeared for either beyond what auto_list already solves.

**Decision — ship A now, hold B as the named backstop.** A is the smallest change that restores the invariant and it uses the mechanism the game designed for exactly this case, so it clears the "fix the producer, smallest change" bar. It beats C because C adds two actions where one optional boolean suffices. It beats B *at this moment* only on sequencing, not on merit: B is genuinely more reliable against a weak planner, but the simplicity rules forbid replacing a mechanism (planner-chooses) with a stronger one (executor-forces) before observing that the cheaper mechanism fails *with the fix applied*. So we ship the briefing and watch.

**Promotion condition (the explicit revert/upgrade trigger).** If, with this auto_list briefing live, the pilot again loops on a "no buyers" sell result — i.e. it receives a no-buyers sell outcome and does NOT plan a follow-up sell with `auto_list=true`, instead re-attempting a plain sell or traveling to another NPC exchange for the same unsellable item — then the briefing alone is proven insufficient and option B is promoted: the executor auto-retries a no-buyers sell once with `auto_list=true` deterministically. That is a failure class recurring after a fix (invariant-promotion trigger: agent/pilot failure → deterministic code), and the deterministic retry then becomes the backstop with the briefing kept as the soft layer.

## 2026-07-12 — Disposing of no-demand cargo: jettison (auto_list falsified live)

**Context — the premise the last fix rested on was false.** The auto_list decision above shipped on an assumption reasoned from the OpenAPI request shape and a one-line wiki note, never watched: that `sell` with `auto_list=true` lists no-demand cargo on the player exchange and frees the hold. Then we watched it. The pilot, holding 10× Palladium Ore that no NPC buys, ran exactly the call PR #75 told it to and got the plain-sell result back — "Sold 0 Palladium Ore for 0cr. 10 unsold (no buyers)." It did not list the ore, escrow it, or free the hold. The planner re-fixated; the SM-9 thrash damper contains the loop (~15 min backoff) but the pilot makes no progress. The invariant is unchanged — after a no-buyers sell the hold must be clearable — but the mechanism we named for clearing it does not work for no-demand goods. "Prove the premise before replacing a mechanism," in reverse.

**Options.**
- **(A) Register `jettison(id, quantity)` — SHIPPED.** Destroys the goods, but a no-demand item's realizable value is already zero. The only escape depending on nothing external (no NPC buyer, no player, no price guess), so the only one that GUARANTEES the unstick. Params are real fixture properties; conformance passes. Receipt for the new registry surface: jettison is a capability a miner wants regardless (you mine junk alongside the good stuff), so it earns its place beyond this one incident.
- **(B) Register `create_sell_order` with a per-unit price — DEFERRED.** The real player-exchange listing (what we wrongly thought auto_list was) and the value-PRESERVING answer. Needs pricing logic we lack — too high never fills, too low gives it away — and a no-demand item may never fill at any price, committing the hold forever. Build it once a captured `view_market` gives us prices.
- **(C) Leave it as dead weight; rely on the thrash damper — REJECTED.** Contained is not solved: the doomed sell is re-planned forever, the blocked ratio stays high, a cargo slot is occupied, and every damped cycle is wall-clock not spent mining something sellable.

**Decision — ship A, name B as the upgrade.** The pilot was trapped precisely because every mechanism it had required demand that did not exist; jettison requires none. For *no-demand* goods this supersedes the auto_list briefing — the digest no longer claims auto_list frees the hold. The parameter stays registered on `sell`: it may still be right for the different, un-falsified case of a partial fill or a temporarily absent buyer.

## 2026-07-12 — A deterministic progress heartbeat: report every cycle, don't act

**Problem.** The operator wanted a continuous, at-a-glance answer to one question: "is the pilot actually making progress right now?" The stall-watcher already exists, but it is an *actor*, not a *reporter* — it stays silent while a pilot is healthy and only speaks when it decides to intervene after a long no-progress window (distress call, re-steer, opt-in self-destruct). Silence from the stall-watcher means "nothing wrong enough to act on yet," which is not the same as a positive "yes, still advancing" signal the operator can watch. There was no steady pulse a human could glance at to confirm the pilot is earning, mining, training, or trading — only the absence of alarms.

**What "progress" means here (SSOT).** The multi-dimensional no-progress detector (`src/agent/no-progress-detector.ts`) already settled the hard question of which counters count as real progress: an allowlist (`PROGRESS_COUNTERS`) of productive-outcome counters (credits earned, ore mined, missions/trades completed, crafting, salvage, kills, scans, discoveries), deliberately excluding movement counters (jumps, distance, systems explored) and the ever-rising clock (time_played), because a pilot that only ever *moves* is wandering, not progressing. The heartbeat must agree with the stall-watcher on this exactly — a divergence would let the dashboard show "progressing" while the stall-watcher counts the same pilot as stalled. So the heartbeat *imports and shares* `PROGRESS_COUNTERS`; it does not redefine it.

**Options.**
- *(A) In-harness deterministic heartbeat [CHOSEN].* Add a per-agent timer, on the same tick loop the stall-watcher and wake cadence already use, that every N minutes computes the per-dimension delta over `PROGRESS_COUNTERS` since the last heartbeat and emits a `progress_heartbeat` event to the existing store — carrying the deltas that changed, a `progressing` vs `stalled` boolean, the window length, and the current position (credits, cargo, system). The dashboard already streams and renders every stored event, so it surfaces with a one-line filter addition and zero new plumbing. Zero LLM cost. It shares the stall-watcher's *counter* definition by construction (same imported allowlist), so the two never disagree on the counter dimension. One honest limit worth recording: the stall-watcher's full progress signal is broader — it also credits skill XP and achievements (its `progressGrandTotal`) — which this heartbeat, a zero-extra-cost pulse that reads only the counter stats already on the current tick, does not. So a rare window whose *only* advance is skill XP or a new achievement (no counter movement) reads `stalled` on the dashboard even though the stall-watcher would not act. The direction that would matter most — the dashboard showing `progressing` while the pilot is truly stalled — is still prevented, because any counter move that flips the heartbeat to `progressing` also moves `progressGrandTotal`; and in practice skill XP almost always rides alongside a counter (mining ore bumps both `ore_mined` and mining XP), so the pure skills-only window is narrow. Aligning the heartbeat to the full multi-dimensional signal (credits OR skills OR achievements) is a named follow-up, deferred only because skills/achievements are sampled on the slower snapshot cadence, not every tick.
- *(B) External host cron script reading the SQLite DB.* A separate process on the deploy host queries the events/stats table on a schedule and reports. Rejected: it is a second deployable to build, run, and monitor, and — worse — it would have to *re-derive* which counters count as progress, duplicating `PROGRESS_COUNTERS` outside the harness where it will drift from the detector the first time either side changes. More moving parts for strictly less consistency.
- *(C) Periodic orchestrator (LLM) check.* Have an LLM look at the pilot every so often and judge progress. Intelligent and able to explain *why* a pilot is stuck, but it costs tokens on every check and is not always-on — exactly the wrong tool for a cheap, continuous, deterministic pulse.

**Decision — ship A, the in-harness deterministic layer.** The harness already owns everything this needs: the progress counters (parsed onto each tick's status), the event store, and the dashboard render path. A is the only option that reuses the stall-watcher's progress definition instead of duplicating it, costs nothing per cycle, and is genuinely always-on. Note the operator chose a *layered* answer, not an either/or: this deterministic heartbeat is the continuous floor, and a periodic orchestrator (LLM) review sits *on top* for the occasional "why" — the deterministic layer is not a replacement for human/LLM judgment, it is the cheap always-on substrate under it.

**Scope boundaries held.** The heartbeat REPORTS only; it never acts. Acting on a stall stays the stall-watcher's job, and the stall-watcher's behavior is unchanged — the heartbeat is a read-only observer on the same loop. Surfacing is dashboard-only by the operator's explicit choice: no notification/push path was built. A healthy pilot now shows a steady stream of `progressing` heartbeats on the telemetry feed; a stalled one shows `stalled:true` every window, visible at a glance.

**Two smaller calls worth recording.**
- *Config knob name.* The natural name `heartbeat_minutes` was already taken — it is the *planner wake* cadence (default 15). Reusing it would have coupled two unrelated cadences. The knob is `progress_heartbeat_minutes` (default 30), distinct from both `heartbeat_minutes` (wake) and `stuck_window_minutes` (the stall-watcher's act-on-stall window).
- *Baseline persistence.* The last-heartbeat baseline lives in memory, not the store. On the rare restart the agent simply re-baselines against the first status it sees and the next heartbeat fires a window later — a one-window gap whose only cost is a single delayed pulse. Persisting it would add a store round-trip whose sole payoff is a seamless window across a process boundary that happens rarely; KISS won.

## 2026-07-12 — Seeing action RESULTS, not just actions: the notification feed + per-tick ledger

**Problem.** The dashboard showed what each pilot DID but not what it GOT. The telemetry feed said "mine" and "sell", but not how much ore came out of the ground, which resource it was, or what a sale actually earned. The operator's question was concrete: per tick, how much ore mined and which resource gained, sell revenue, gains and losses — the outcomes, not the verbs. Three facts about the harness made this harder than "print the result string." First, action events already carry a `result` string and the dashboard rendered it, but truncated to 80 characters, so "Sold 2 Carbon Ore for 4cr" was often cut off. Second, and worse, `mine` doesn't return its yield at all: its action result is "Action pending. Resolves next tick" — the amount and the resource land a tick LATER, in the status snapshot, nowhere near the action that caused them. Third, the game emits its own NOTIFICATIONS (the result feed: mining yields, sales, combat), which the loop already fetches every tick to feed the planner, but which were never turned into dashboard events.

**What "the answer" has to survive.** Because a mine's yield resolves asynchronously, any approach that reads only the action's own result string is structurally blind to the single outcome the operator most wanted (how much ore, which resource). The reliable signal has to come from somewhere that sees the yield after it lands — either the game's notification for it, or the change in the ship's cargo between ticks. Both are data the loop already has in hand each tick; neither costs an extra game call.

**Options.**
- *(A) Emit game notifications + a status-delta ledger [SHIPPED].* Two complementary observers, both pure retain-and-expose over data the loop already fetches. (1) Turn each new game notification into a `notification` event, deduped by id so the same one isn't restamped on every poll (the game keeps returning the recent batch until entries age out). This surfaces the game's OWN result feed verbatim — whatever it reports, yields/sales/combat. (2) Diff this tick's status against the previous and emit a `ledger` event capturing the deltas: a cargo item's quantity rising ("+3 Carbon Ore, cargo 8->11"), credits moving ("+4cr" income, "-186cr" spend). Emit only when something changed, so a quiet tick is silent. The ledger is the deterministic backbone: it depends on nothing but the status the loop already reads, so it answers "how much ore, which resource, what revenue" even for the async mine yield and even if a notification is missed or malformed. Plus the trivial dashboard fixes: widen the action-result display from 80 to 160 chars, and render the two new event types (default-visible, since they're low-volume — unlike status_snapshot).
- *(B) Parse each action's result string into structured outcomes.* Regex/parse "Sold 2 Carbon Ore for 4cr" and friends into fields. Rejected as both fragile and incomplete. Fragile: it couples the harness to the game's exact human-readable result wording, which can change without notice and varies per action. Incomplete: it cannot answer the mine-yield question AT ALL, because the mine result string is "resolves next tick" — the yield is never in the string to parse. It would be real work to build the exact thing that structurally can't see the most-wanted outcome.
- *(C) A richer structured stats panel (cumulative totals per resource).* Show lifetime ore-per-type, total revenue, etc., derived from the stats block. Rejected for THIS need as the wrong axis: cumulative totals answer "how much in total" but bury "what happened THIS tick" — the operator asked for the per-tick stream, the gains and losses as they occur. A totals panel is a fine SEPARATE addition (and the progress heartbeat already reports per-window deltas over the productive counters), but it doesn't give the moment-to-moment result feed the operator wanted, and it can't attribute a specific yield to the tick it landed on.

**Decision — ship A.** The status-delta ledger is the honest answer to the operator's exact question because it is derived from the one source that sees every outcome including the async mine yield: the change in ship state between ticks. It costs nothing extra (the status is already fetched), it can't be defeated by result-string wording changes (B's fragility) or by the yield being absent from the string (B's blind spot), and it stays on the per-tick axis the operator asked for (C's mismatch). Pairing it with the raw notification feed gives the operator both the game's own words for what happened AND a deterministic delta they can trust when the words are missing. The two never double-count: the ledger is strictly a DELTA, while the absolute series lives in status_snapshot (which stays hidden by default). We accept one honest limit — the ledger reports cargo and credit movement, the two dimensions the operator named; a yield that shows only in a lifetime stat but never touches cargo or credits (none of the productive outcomes do) would be invisible to it, which is why the notification feed rides alongside as the game's own catch-all.

**Improv-mode note (harness observability backstop).** The notification feed and the per-tick ledger are model-independent observability, not pilot behavior — they change nothing about what any pilot decides or does, so they need no improv-mode briefing rule. They are a deterministic backstop that stays deterministic: whatever drives the pilot (scripted planner, cheap-tier model, or a fully self-driving improv agent), the operator sees the same truthful per-tick result feed and credit/cargo deltas, because both are derived by the harness from game state the loop already fetches, entirely outside the planner's control.

(Plan: `docs/superpowers/plans/2026-07-12-improv-mode-implementation.md`; §5's unknowns now point at the fixture, §5a documents the dashboard format.)

## 2026-07-12 — Batch A: how to parse the MCP text dashboards (and where the transport lives)

**Context.** Batch 0 proved the MCP path returns a human-readable text dashboard, not the structured object our HTTP client Zod-parses — yet every backstop consumes a typed `StatusSnapshot`. The format is line-oriented, mixes pipe/tab delimiters, uses thousands separators, and — the real trap — its row order is NOT stable between calls.

**Options.**
- **(A) Line-oriented hand parser** — a state machine walking the document top to bottom. Faithful, but couples the parse to section ORDER and row counts, so a moved section or shifted header throws the walk off. REJECTED.
- **(B) Regex-per-field for everything** — order-immune and trivial for scalars, but it cannot associate "22" with "Palladium Ore" across a row, and silently mis-keys when row order changes. REJECTED as a whole-file strategy.
- **(C) One general table tokenizer for everything** — over-fits the other way: the header/ship/capacity lines are pipe-delimited `Label: value` segments, so it means inventing fake single-row tables. REJECTED.

**Decision — a hybrid, because the two halves differ.** Scalars (fuel, cargo, hull, credits, docked-at) use per-field regexes over the whole text: one-liners, immune to section order. Tables (cargo, POIs, connections) use a tokenizer that reads the table's own header row and keys every cell by COLUMN NAME, not position. That one choice defeats both proven traps: unstable row order is irrelevant (callers diff by id, never index), and a column reorder cannot misalign a cell. `split("\t")` keeps empty cells, so a blank faction/class column does not shift a row; numbers are comma-stripped before parsing.

**What the parser refuses to invent.** `systemId` is `null` (the status header names the system, not its id; `get_system` carries it) — the improv stuck-fingerprint's system dimension is weaker, though `dockedAt` keeps it from going blind. Lifetime `stats` is `undefined`, which the no-progress detector already treats as UNKNOWN and suppresses. One derivation IS made: cargo rows give display names, not ids, so the parser slugifies. Not a guess — a fixture-oracle test asserts `slug(name) === item_id` for every captured cargo item, holding across hundreds of market rows ("Helium-3" → `helium_3`).

**Smaller call — the transport is a sibling module, not a mode flag on `SpacemoltHttp`.** The wire formats diverge too far for a shared body to be anything but conditionals: JSON-RPC vs REST, SSE vs plain JSON, session-as-argument vs `X-Session-Id` header. What maps IS reused: the retry loop, `MAX_RATE_RETRIES`, and `SpacemoltError` from `http.ts`, and the probe's JSON-RPC builders are composed, not re-derived.

**The honest caveat carried in the code.** The `-32001` session-recovery path (single-flight renewal + sliding-window breaker) and the `429`/Retry-After handling are tagged **load-bearing: ASSUMED** in `mcp-errors.ts` and `mcp.ts` — a read-only probe cannot provoke an expiry or a rate-limit, so both are built from the two agreeing references, not ablated against our own account. The tag drops the first time a live window hits one and we capture the real shape.

## 2026-07-12 — Pacing repeated actions to the game tick (stop re-firing a resolving action)

**Context.** The dashboard filled with waste: a mining pilot re-attempting the same action several times inside one ~10s tick, producing "Another action is already in progress" and a stream of "Action pending. Resolves next tick." That pending text is an ACCEPT, not a result — the yield lands a tick later. Loop and tick are both ~10s but unsynchronised, and the executor treated a pending accept as an ordinary `continue`, re-firing the same step next loop. Nothing BROKE (the SM-10 transient-block guard catches the rejection as a `wait`), but every racing loop burns a submission and stamps the feed. No busy flag exists for non-movement actions (only `in_transit` for travel), so the executor had no pre-check. Invariant: an accepted action still resolving must not be re-submitted until its tick resolves, and the bad state is produced at the re-fire decision.

**Options.**
- **(A) Tick-pacing settle — SHIPPED.** On a "Resolves next tick" accept where the plan holds the same step, the executor flags the `continue` with `settle`: one tick passes without a submission, so submissions and settle ticks alternate. Receipt for the new state — one boolean (`awaitingResolution`) — the tick boundary falls between loop iterations, and the executor is stateless across them.
- **(B) Reuse the existing `wait` outcome — REJECTED.** `wait` re-submits on the very next loop, re-entering the identical race: it relabels the churn without reducing it, and the operator asked for less waste, not a renamed event.
- **(C) Widen the loop interval to 15s — REJECTED.** Reduces the race only probabilistically, slows every action including synchronous ones, and hard-codes a guess about tick length instead of reading the game's own signal.
- **(D) Poll status for a cargo/credits delta — REJECTED.** Rebuilds a detector the game gives us free, and reopens a cache-input problem (is a zero-yield mine tick resolved?) where a false "resolved" re-races and a false "pending" holds forever.

**Decision — ship A**, scoped to the same-step re-fire actually diagnosed; the cross-step case is rarer and already covered by the transient-block `wait`. Not a permanent hold: the settle is one-shot, `lastPlanAt` is written on neither fire nor settle, and `awaitingResolution` clears on every replan, so the heartbeat wake and the no-progress detector both still escalate. Paired improv briefing rule recorded in the improv spec (§4 timing, §5 backstop).

## 2026-07-12 — Ingest the game's catalog.json as the item-economics SSOT (#104)

**Problem.** The pilot dumped palladium_ore as junk. Palladium is a base_value-200 rare ore that feeds five recipes (superconductor, focused crystal, sensor array), but the harness had no reference for what any item is worth or whether anything crafts from it, so a "sell" that found no local buyer read to the loop as "worthless, discard." The game already publishes the answer: a static, public, auth-free catalog at `https://game.spacemolt.com/api/catalog.json` — 710 items with `base_value`/`rarity`/`tradeable`/`extracted_by`, and 721 recipes with inputs and outputs. We simply were not reading it. The task was to make that catalog the single source of truth for item economics that downstream jettison-gating (#94), sell/haul, and view_market (#93) decisions read from, so those decisions are data-driven rather than reactions to one market's silence.

**How the catalog reaches the harness — three options.**
- *(A) Vendor a trimmed snapshot into the repo + a schema-tolerant loader [SHIPPED].* Fetch the catalog once, trim it to the fields we use, commit it as `data/catalog.json`, and read it at process start into an in-memory index. The item DB is a slowly-changing reference table, not live state — an ore's base_value and a recipe's inputs don't move tick-to-tick — so it belongs in the build, versioned and reviewable, exactly like the OpenAPI slim fixture. Cost: the snapshot goes stale and needs an occasional refresh (a re-fetch + re-commit), which we accept because staleness here is visible in git and cheap to correct, and because a committed snapshot is the only form that keeps tests fully offline (AGENTS.md: zero live-game traffic in tests).
- *(B) Fetch the catalog live at runtime (on boot, or on a timer).* Always current, no refresh chore. Rejected as the wrong tradeoff for a reference table. It puts a network dependency on the boot path of a cost-minimizing harness — a 5 MB download the game has to serve and we have to tolerate failing — for data that changes on the cadence of game patches, not gameplay. It also can't be tested offline without mocking the fetch, which just reintroduces a fixture by the back door. The right home for "current" is a deliberate refresh of the vendored snapshot, not a boot-time call.
- *(C) Hardcode the handful of values the pilot tripped over (palladium and friends) in code.* Smallest possible change, zero new files. Rejected because it doesn't build the thing the backlog needs — #94, #93, and crafting all want to ask "what is item X worth / is X a recipe input" for arbitrary X, not for a curated shortlist. Hardcoding palladium fixes the one incident and guarantees the next unlisted ore repeats it. The catalog is the SSOT precisely so we stop patching item economics one incident at a time.

**Decision — ship A: vendored snapshot + schema-tolerant loader.** The snapshot lives at `data/catalog.json` (a new `data/` dir for runtime reference data, distinct from `test/fixtures/` which is test-only inputs); the loader is `src/catalog/catalog.ts`; both the loader and the tests read the one file (SSOT, no duplicated fixture). The raw catalog is 5 MB because it also carries ships, skills, facilities, achievements, and per-item equipment stats we don't use here; the vendored copy is trimmed to 460 KB by keeping only `items[].{id,name,category,base_value,rarity,tradeable,extracted_by}` and `recipes[].{id,name,inputs,outputs}`. The trim is faithful (no values altered) and recorded in the file's own `note`/`fetched`/`source`/`version` fields (snapshot version `0.493.0`, fetched 2026-07-12).

**Why the loader is schema-tolerant, and what that means concretely.** The catalog will evolve — fields get renamed, added, or dropped across game patches, and a committed snapshot outlives the schema that wrote it (the same persisted-state-tolerance rule that the 2026-07-12 chat-enum incident promoted to a binding convention). So the loader is built to degrade, never crash: every field except an entry's `id` is optional and individually caught, so a mistyped or renamed `base_value` nulls that one field rather than dropping the item; unknown fields are ignored; an entry with no usable string `id` is skipped; and a catalog whose top-level `items`/`recipes` are missing or non-arrays yields an empty index instead of throwing. The API the backlog consumes is four functions bound to the default snapshot — `itemValue(id) -> number | undefined`, `itemMeta(id) -> {name,category,base_value,rarity,tradeable,extracted_by} | undefined`, `isRecipeInput(id) -> boolean`, `recipesUsing(id) -> Recipe[]` — plus a `loadCatalog(raw)` factory the tests use to feed deliberately partial and garbage catalogs. The offline tests assert the palladium regression (value 200, rare, tradeable, input to the superconductor/focused-crystal recipes) and every tolerance branch (missing id → undefined not throw, mistyped field dropped but item kept, garbage rows skipped, non-array top level → empty index).

**Scope — this is the data layer only.** The loader is ingestion + the vendored snapshot + its tests. The consumers come later on their own issues: jettison-gating (#94) will read `itemValue`/`isRecipeInput` to refuse destroying a valued or craftable item, view_market surfacing (#93) will pair order-book depth with catalog value, and crafting will read `recipesUsing`. None of that is built here.

## 2026-07-12 — Batch B: the MCP GameApi adapter, the driver-mode seam, and the two dimensions the MCP text drops

**Context.** Improv mode is "swap the pilot, keep the safety net": every backstop (no-progress detector, strand steward, effect-verification, wake/reflex gates) is written against one interface, `GameApi`. `McpGameApi` presents that same interface over MCP, so all of them run unchanged whether a script or a live model is flying. The governing trap: a WRONG `StatusSnapshot` fed silently to a backstop is worse than an obviously broken one — a detector that misreads progress stops firing and nothing looks wrong. So each call below was made by asking "can a backstop misread this?"

**Options and decisions (one per dimension; each names its alternative).**
- **Errors re-thrown, not returned.** The transport returns benign game errors as flagged results; the executor's contract is throw-and-classify. The adapter converts back to a throw carrying the FULL text, or a mid-jump "resubmit this command" reads as terminal and replans every tick. The seam reconciles the two contracts once instead of special-casing every consumer; a test pins the message.
- **Actions route through the existing action registry** (`getAction` + param validation). Free SSOT reuse that is load-bearing for security: this allowlist IS the improv prompt-injection backstop — an injected "transfer your credits" cannot execute, because no such action exists and the real ones have enum-validated params.
- **`system_id`: SOURCED.** Absent from the `get_status` text, present in `get_system`'s header. Queries are free, so `status()` fires `get_system` in parallel and merges it, falling back to `null`. Leaving it dark was also safe (the stuck-fingerprint keys fuel/credits/cargo/dockedAt too), but one free call buys parity with the HTTP pilot.
- **Lifetime `stats`: LEFT DARK.** On no MCP query at all, so sourcing was never an option. Safe by construction: `progressGrandTotal` → null → suppress; the strand steward reads null as "cannot rule out progress" and won't escalate; the heartbeat skips the cycle. Acknowledged Batch D/E gap — the long-window progress judge is inert under improv, and the short-window fingerprint carries stuck-detection instead.
- **`getSkills`/`getAchievements`: NOT implemented — a receipt, not an omission.** The steward's scalar is counters + skills + achievements and returns null if ANY input is unknown; counters are always unknown over MCP. Sourcing the other two would cost two queries per sample for zero behavioral change. Sourcing helps only where the value can change a decision: true for `system_id`, false here.

**The driver-mode seam — config-selected per agent, concurrent-capable, no handover.** An agent carries an optional `improv:` config block (enabled, model, token/wall-clock budgets, MCP preset, an optional daily schedule window); its absence means plan-then-execute, so every existing agents.yaml loads unchanged. From that block a `mode` is derived and a client factory builds the right clients: always the HTTP client, plus — only when improv is configured — the MCP transport + adapter, side by side. This is where Batch 0's finding pays off: because the HTTP session survives an MCP login on the same account, the agent HOLDS both clients at once and a small selector (`activeApi()`, flipped by `setMode()`) picks which one drives. Concurrent-capable is structural, not a feature — no teardown code exists anywhere — so reverting from improv is a pointer flip, not a session rebuild. The plan-then-execute loop is untouched: a default-mode agent behaves byte-identically whether or not an improv client is also wired (a test asserts exactly that). The MCP session is built dormant and logs in only when a supervised window opens (Batch 0's at-login caveat).

**What Batch B deliberately did NOT build (scope discipline).** No improv loop, chooser, or briefing (Batch C); no improv-specific backstops or game-state-only fingerprint (Batch D); no window/schedule triggers, reversion plumbing, or usage metering (Batch E). Batch B is the adapter plus the seam the later batches plug into — the mode selector exists and is tested, but until Batch C wires the loop, an improv-configured agent still runs plan-then-execute.

## 2026-07-12 — Base-earning pivot #1 (#124): register the mission flow + market-finders, brief the planner

**Problem.** Inspecting the live pilot surfaced a plain structural gap: `missions_completed` has sat at 0 forever, not because the pilot chose to ignore missions but because it *could not touch them* — no mission actions were ever registered. A mission is the game's ~10x earner (it pays far more than dumping ore on an exchange), and the pilot had no way to list one, accept one, or complete one. The same inspection showed the twin gap on the sell side: the pilot kept hitting a local exchange's "no buyers" and stalling, because it had no way to find *where* its ore actually sells. The game exposes all of this; we simply never put it in the registry, so the planner's whole vocabulary for the primary income path was missing.

**Options.**
- *Register the actions + brief the planner now; defer parsing their responses.* Add the five mission actions (`get_missions`, `get_active_missions`, `accept_mission`, `complete_mission`, `abandon_mission`) and the two market-finders (`analyze_market`, `view_orders`) to the registry, request shapes verified against the OpenAPI fixture, and add digest briefing lines that tell the planner missions are the primary income and that `view_orders`/`analyze_market` find a buyer for an item at any station. The planner reads the raw query result text; no deterministic parser of the responses ships yet.
- *Full deterministic surfacing in the same change.* Also parse the `get_missions`/`analyze_market`/`view_orders` **responses** — render a "missions you can fulfill" digest section and a "stations that buy your ore" list, gate the sell/mission preconditions to them. Strongest end state, but it needs the response shapes, and our slim OpenAPI fixture is request-only (it never recorded response bodies). Building the parser now means guessing field names — the exact bug class this project forbids (how the `find_route` and chat-channel bugs happened, and why the `view_market` surfacing was already deferred once). Blocked on capturing the real responses from the live pilot.
- *Do nothing / keep grinding ore.* Rejected on its face: the pilot is structurally locked out of its best income and stalls on ore it can't locally sell. This is the #1 unlock of the base-earning pivot, not an optimization.

**Decision — register + brief now, defer parse-and-surface.** The first option is the honest, zero-unknowns change and it is the whole of issue #124's #1 ask. Registering the seven actions (mission request shapes verified: `accept_mission` props `[id, template_id]` required `[]`, `complete_mission`/`abandon_mission` required `[id]`; finders verified: `analyze_market` props `[]`, `view_orders` props `[item_id, order_type, page, page_size, scope, search, sort_by, station_id]` required `[]`) gives the planner the vocabulary it was missing, enforced by the registry allowlist exactly like every other action. Briefing the planner turns the capability into behavior: missions are the primary income (run `get_missions` → `accept_mission` → work the objective → `complete_mission` when docked), and to sell an item, use `view_orders`/`analyze_market` to find a station with a standing BUY order rather than dumping on the first "no buyers." That last line also refined the existing jettison guidance: `view_orders` is now the cheap check that gates jettison-as-last-resort — look for a buyer anywhere first, dump only when none exists reachable, rather than the old blanket "no buyers here probably means nowhere."

We deliberately do **not** parse or surface the `get_missions`/`analyze_market`/`view_orders` responses yet. Those response shapes are uncaptured (the fixture is request-only), and guessing their fields is precisely the failure this project refuses to repeat. The follow-up is the same recipe used for `view_market`: capture the real responses from the live pilot, then build a deterministic "missions you can fulfill / stations that buy your ore" surfacing against a real fixture — no guessed field names. Register-and-brief is the smallest change that unlocks the income path; parse-and-surface waits on evidence.

## 2026-07-12 — accept_mission needs a template_id: the registered shape was conformant but the game is stricter at runtime

**Problem.** The mission flow from pivot #1 (above) deployed and the pilot started reaching for missions — the intended behavior. But it called `accept_mission` with EMPTY params and the game answered `invalid_payload: Must provide template_id (or mission_id) to accept a mission.` Caught live. The root cause is a gap between two truths that both looked fine on their own: our slim OpenAPI fixture recorded `accept_mission` with properties `[id, template_id]` and required `[]` — so the registry marks both params optional, and a request carrying neither is a *conformant* request by the registered schema. The game, however, enforces "at least one of id/template_id" at RUNTIME, a constraint the request schema never expressed. The planner, seeing both params optional and having no get_missions result threaded into the accept step, emitted the empty call. This is the same lesson we keep relearning: check preconditions deterministically, don't rely on the model remembering the sequence.

**Options.**
- *Brief the planner only.* Tighten the digest so the model always threads a template_id from get_missions before accepting. Necessary — the planner is where the empty call originates — but a briefing is a probabilistic steer, not a guarantee; a cheap-tier planner can still emit an empty accept, and each one is a wasted doomed submission plus dashboard noise.
- *Deterministic guard only.* Refuse the empty-param `accept_mission` in the executor before it reaches the wire. Guarantees no doomed call ships, but on its own it just converts every empty accept into a blocked wake → replan (an LLM call) without teaching the planner to get it right the first time.
- *Both — guard as the backstop, briefing as the producer-side fix.* The pattern every prior precondition fix here already follows (undock-while-undocked, mine/travel in-transit, sell-effect verification): the producer-side steer lives in the digest so the planner picks right, and a deterministic guard is the non-negotiable backstop so a planner miss never spends the doomed call.

**Decision — both.** The guard (`src/agent/executor.ts`) short-circuits an `accept_mission` step to a `blocked` result with reason "accept_mission needs a template_id (or id) from get_missions first" when BOTH `id` and `template_id` are absent or empty strings — mirroring the existing undock precondition guard, and fail-safe by construction: a present `id` OR `template_id` passes straight through to the API untouched, so the guard can only ever stop a call the game was already guaranteed to reject. The briefing (`src/planner/digest.ts`) tightens the existing missions runbook line (from #124) rather than adding a second one: it now spells out that `accept_mission` REQUIRES a template_id (or mission_id) copied from a get_missions entry, forbids empty params explicitly, and names the ordered sequence get_missions → read a template_id → accept_mission(template_id) → work the objective → complete_mission(id).

The transferable lesson: a request shape that CONFORMS to the registered/OpenAPI schema can still violate a stricter runtime rule the schema never encoded. The registry allowlist validates structure, not the game's business rules — so a game-enforced "at least one of X/Y" precondition has to be guarded deterministically at the executor seam and taught at the planner seam, not assumed to be covered because the request validated. Same shape as the `find_route` and chat-channel fixes: the registered contract is necessary but not sufficient, and the ground truth only shows up live.
## Transport resilience: port gantry's error taxonomy + retry, not its framework (2026-07-12)

**Context.** Our SpacemoltHttp client already re-logged in on a narrow set of session
errors and retried rate-limits, but it threw immediately on server-side (5xx) and network
failures and had no protection against hammering a down server. The gantry project
(geleynse/gantry, MIT) — another SpaceMolt client, but over the MCP/JSON-RPC transport, not our
HTTP /api/v2 — had already solved this: a SESSION_EXPIRED / SERVER_ERROR code taxonomy, transparent
re-login, a retry policy with exponential backoff, and a circuit breaker. The interface evaluation
flagged it as a high-value borrow.

**Options.**
- *Copy gantry's modules wholesale.* It ships `circuit-breaker.ts` (271 lines: half-open probe
  state machine, listener callbacks, a BreakerRegistry, aggregate-status reporting) and
  `retry-policy.ts` as standalone files. Rejected: most of that surface exists to feed gantry's
  health dashboard and its WebSocket connection lifecycle — neither of which we have. It's
  MCP/Express-shaped code; porting it verbatim would drag in concepts (WS reconnect, per-agent
  breaker registry) that don't map to a stateless HTTP call. Violates our simplicity rules
  (correct-but-larger = revise) and adds dead flexibility.
- *Port the LOGIC and the error-code taxonomy only, inline, adapted to HTTP.* Take the two things
  that are genuinely transport-independent — the SESSION_EXPIRED_CODES / SERVER_ERROR_CODES
  classification and the backoff math — and re-express them in ~140 lines inside our existing
  `http.ts`, with a minimal per-tool breaker (a failure counter + a cooldown timestamp) instead of
  the full class.
- *Do nothing / hand-roll from scratch.* Rejected: gantry's taxonomy is battle-tested against the
  live game (it carries codes we'd otherwise discover the hard way), and reinventing the backoff
  loop earns nothing.

**Decision — port the logic and taxonomy, not the framework.** The value in gantry here is the
knowledge (which codes mean "session gone, re-login" vs "server sick, back off" vs "your command was
wrong"), not the code. We took that and the exponential-backoff-with-jitter shape and left the
dashboard/WS machinery behind. The circuit breaker is deliberately minimal: per-tool so one broken
action can't blind the whole client, opening after repeated server failures and failing fast for a
30s cooldown — no half-open probe or registry until a dashboard actually needs live breaker state
(the upgrade path is noted in a `ponytail:` comment). One correctness fix landed in review: all 5xx
(not just 502/503/504) and any non-JSON error body are now transient, because a bodyless HTTP 500
would otherwise throw a raw SyntaxError past the classifier — the exact failure the port set out to
tame. Offline tests cover transparent re-login on the broadened taxonomy (code and message-based),
backoff-then-succeed, retry exhaustion, the bodyless-5xx/non-JSON paths, and the breaker's
open/fail-fast/recover-after-cooldown cycle. This is a §5 deterministic backstop under improv mode
(see the improv-mode spec) — the model never sees the retries, only the outcome.
