## 2026-07-10 — The project itself: SpaceMolt as an agent-orchestration lab

**Concept.** SpaceMolt is an MMO (massively multiplayer online game) where the players are AI agents, not humans. It exposes a plain HTTP API and runs the whole universe on a 10-second heartbeat (a "tick"), like a game-flavored message queue: you submit one state-changing action per tick, read-only queries are free and unlimited. That makes it an ideal lab for the real goal — learning to run a fleet of AI agents with monitoring, config management, and feedback loops. Think of it as a staging environment where outages are entertaining instead of career-limiting.

**Why this way.** The game gives us everything a real automation platform needs to practice against: long-running processes, an external API with rate limits and session expiry, observable outcomes (credits earned, ships lost), and a reason to build dashboards.

**Tradeoffs.** It's a hobby game server we don't control — the API can change under us (we build a drift-detection test for exactly this), and uptime is best-effort.

## 2026-07-10 — Plan-then-execute: the LLM is a planner, not a pilot

**Problem.** An agent has to decide what to do on every 10-second tick, and the expensive part — the LLM — costs on every call. How often should the model actually be consulted?

**Options.**
- *Model-in-the-loop (the naive design):* ask the LLM "what should I do?" every tick, like calling a consultant every time cron fires. Maximally responsive — the agent reconsiders constantly and never misses a shift in the world. But it costs ~360 LLM invocations per hour per agent, which for a cost-first project is disqualifying on its own.
- *Plan-then-execute:* the LLM writes a short *plan* (a runbook: "fly to the asteroid belt, mine until the hold is full, dock, sell everything") and a dumb deterministic executor — ordinary code, zero AI — walks it step by step. The model is only re-called ("woken") on events a runbook can't handle: the plan finished, a step failed, the ship is under attack, fuel is low, the operator typed an instruction, or a 15-minute dead-man timer expired. Costs 4–10 invocations per hour. The price is that an agent mid-runbook is blind to subtle opportunities it wasn't told to watch for — it won't notice a market spike unless something wakes it.

**Decision.** Plan-then-execute, on cost — a ~40-to-1 reduction is the whole ballgame for a project whose first-class goal is minimizing spend. It's also the same pattern as any good automation: humans (or LLMs) write the playbook, machines execute it, escalation paths handle the exceptions. We accepted the blindness for miners and explorers and bought some of it back by giving the combat agent more trigger-happy wake conditions. The wake-condition list is effectively our alerting rules, and like all alerting rules, it will need tuning.

## 2026-07-10 — Claude subscription + local Ollama instead of API keys

**Problem.** LLM (large language model) providers bill two ways: metered API keys (pay per token, like cloud egress) or flat-rate subscriptions (like a leased line). For a harness whose planner runs continuously, which billing model — and which provider — keeps marginal cost near zero?

**Options.**
- *Metered API keys:* simplest to wire and fully headless-friendly, but every token is billed, so a chatty agent is a growing invoice. Kept in the design as an off-by-default escape hatch, not the default path.
- *Claude subscription via headless mode* (`claude -p`: no graphical interface, just text in and out, so it scripts like any CLI tool): rides the flat-rate subscription the user already pays for, so planner calls add no marginal cost. The catch is a 5-hour rolling usage cap — hit it and calls fail rather than costing more — and headless Claude in a container needs writable auth state (OAuth, the standard login-and-permission handshake, refreshes tokens every run), which is unproven enough to gate the build.
- *Local Ollama* (open-weight models — parameters you can download and run yourself — on our own hardware): free at the margin like self-hosting instead of SaaS; good for routine replans, weaker than the frontier models for hard judgment.

**Decision.** Use the Claude subscription for planner calls and local Ollama for routine replans, with metered API keys as the off-by-default fallback — zero marginal spend, which is the point. The subscription cap makes "subscription limit reached" a first-class failure mode with its own handling (fall back to the local model or cool down until the window resets). The container-auth question is real enough that it becomes phase 0: a spike that gates the rest of the build.

## 2026-07-10 — One process, one container

**Problem.** Three agents plus a web dashboard need to run somewhere. Split them into separate services (the "proper" distributed answer) or bundle them into one deployable unit?

**Options.**
- *Distributed (one container per agent + shared Redis/DB + separate dashboard service):* the textbook distributed answer, and the right one at scale — independent restarts, fault isolation. But three agents don't need service discovery (the machinery that lets separate services find and talk over a network), and the supporting cast (a message bus for passing messages between services, a shared store, service-to-service networking) is all overhead you run and debug for isolation you don't yet need.
- *Monolith (all three agents + dashboard in one process, one container):* one log stream, one restart, one compose entry, and in-memory function calls where the distributed design would need a message bus. Simplest thing that works. Costs: can't restart one agent without restarting all, and one crash takes everything down.

**Decision.** The monolith, deliberately — at three agents the distributed design buys isolation we don't need and adds machinery we'd have to run and debug. The "can't restart one agent" cost is accepted for v1 (a dashboard pause button is the v2 fix); the "one crash kills all" cost is mitigated by SQLite crash recovery (below). If agent count ever grows past a handful, this decision gets revisited.

## 2026-07-10 — Polling HTTP instead of WebSockets

**Problem.** The game offers two transports: real-time push (WebSocket) and plain request/response (HTTP). Which does the loop use to learn about new events?

**Options.**
- *WebSocket push:* sub-second latency on chat and combat events, but carries a reconnect state machine and is harder to debug — you can't curl a persistent socket by hand or replay it.
- *HTTP polling:* each loop iteration asks "anything new?" alongside its normal work. Every exchange is a discrete request you can log, replay, and reproduce by hand, with no reconnect logic. The cost is that events arrive up to one loop-iteration late.

**Decision.** Poll over HTTP. The game only accepts one action per 10-second tick, so sub-second push latency buys nothing — the same reasoning as choosing a cron poll over a webhook when the data only changes every 10 seconds. The one-iteration lag on chat/combat events is acceptable; the spec keeps WebSocket as a v2 option if polling ever proves limiting.

## 2026-07-10 — SQLite as the only state store

**Problem.** Agents produce a stream of events (decisions, actions, errors), and each carries state (current plan, which step) that must survive a crash. What stores it?

**Options.**
- *A database server (Postgres, plus maybe Redis for state/cache):* the conventional answer, but every server is another container to run, back up, and keep alive — the exact infrastructure the monolith decision was made to avoid.
- *SQLite (a single-file embedded database, no server process):* zero extra infrastructure — backup is copying one file on a volume. Single-writer by design, which is fine for one process but a rewrite if agents ever become separate processes.

**Decision.** SQLite, as the *only* state store — it matches the monolith: one process, one writer, one file. The event log then does triple duty: dashboard history, debugging record, and crash recovery. On restart each agent reloads its plan and *cursor* (which step, which iteration — updated transactionally on every step) and resumes exactly where it stopped, never re-running a completed step (which matters: replaying a "sell" would double-sell). There is no separate save system to keep consistent with the log. Events are pruned after 30 days to cap disk growth.

## 2026-07-10 — A single action registry (SSOT applied)

**Concept.** Every game action the agents can take (mine, travel, sell...) is defined in exactly one table in the code, with its parameters and validation. Three consumers derive from that one table: the schema (structured template describing what valid data looks like) the LLM's plans are validated against, the executor's vocabulary, and the dashboard's event labels. This is SSOT (single source of truth — one authoritative copy of a fact, everything else derives from it), same principle as defining a firewall rule in one Terraform module instead of hand-copying it into three environments.

**Why this way.** When the definition lives in one place, the LLM cannot plan an action the executor can't run, and the dashboard can't mislabel it. Drift between copies is the classic config-management failure; we removed the copies.

**Tradeoffs.** The registry is a hand-curated subset (~16 of the game's 100+ commands) — agents can't use what we haven't registered. A CI test cross-checks every registry entry against the game's published API spec, so if the game changes an action signature, a test fails instead of an agent silently breaking at 3am.

**Alternatives: none seriously considered.** SSOT is a binding project rule, so there was no fork here — the alternative (hand-copied action definitions in the schema, the executor, and the dashboard) is the exact config-drift failure the rule exists to prevent.

## 2026-07-10 — Zero-token testing with a fake game server

**Problem.** The agent loop has to be tested — login, planning, executing, session expiry mid-plan, crash recovery. What does it run against?

**Options.**
- *The live game and a real LLM:* maximally faithful, but slow, flaky, rate-limited, and it spends real subscription usage on every run — like load-testing against production. It also can't script the failure cases (session expiry at an exact step) the loop most needs to prove.
- *A fake server + scripted planner:* a miniature in-process imitation of the game server (canned responses, scriptable failures) and a scripted stand-in for the LLM. The full loop runs end-to-end in milliseconds with no network and no LLM calls. The catch: the fake is only as accurate as our reading of the API docs.

**Decision.** The fake server is the lab bench — zero-token, offline, deterministic, and able to script exactly the failure paths that matter. The accuracy gap is narrowed two ways: the conformance test (above) checks request shapes against the published spec, and the first live run (Plan 4) is the real validation. LLM decision *quality* is untestable by definition here — that's what the dashboard is for.

## 2026-07-10 — Bun runtime, TypeScript, near-zero dependencies

**Problem.** The harness needs a runtime to run the TypeScript. Which one?

**Options.**
- *Node.js:* the mature, ubiquitous default, but it needs a build step for TypeScript and pulls in separate packages for a test runner, a SQLite driver, and a YAML parser — every one a dependency to install, patch, and audit.
- *Bun (a Node.js competitor):* runs TypeScript directly with no build step and ships batteries-included — built-in test runner, SQLite driver, YAML parser, single-binary compile. Younger than Node, so occasional ecosystem rough edges.

**Decision.** Bun, for two reasons. The game's own reference client is Bun, so we can borrow proven session-handling code; and every built-in is a dependency we don't install, shrinking the supply-chain surface to a single package — Zod, which checks at runtime that data actually matches its intended shape (schema validation). Our needs are plain HTTP, SQLite, and tests, all first-party Bun features, so the maturity gap costs us little.

## 2026-07-10 — Independent review agents, never self-review

**Concept.** Every artifact (spec, plan, soon code) is reviewed by a *fresh* AI agent with no involvement in writing it, before the user sees it. The author context never grades its own homework.

**Why this way.** An authoring context re-reads its own assumptions as facts — same reason the person who wrote the change doesn't approve their own PR. Empirically it works: the spec review caught a genuine blocker (read-only auth mounts break token refresh), and the plan review caught a test that would have failed and an alert rule that could never fire (`player_died` arrived under a category our filter excluded).

**Tradeoffs.** Each review costs a subagent run and wall-clock time. Worth it: every catch so far would have cost more to discover during execution.

**Alternatives: none seriously considered.** Self-review by the authoring context is precisely the thing this rule forbids — a context re-reads its own assumptions as facts — so there was no second option, only the question of whether to pay for a fresh reviewer, which the empirical catches answered.

## 2026-07-10 — Subagent-driven execution

**Concept.** Implementation work is dispatched to fresh subagents task-by-task; the main conversation context acts as the orchestrator — dispatching, reviewing results, maintaining this log, and talking to the operator. Analogy: the main context is the CI controller, subagents are ephemeral build workers.

**Why this way.** User directive. A context window is a finite resource like RAM; filling it with implementation detail (file contents, test output, compiler errors) degrades the orchestrator's judgment and burns tokens re-reading state. Workers get exactly one task's context, do the work, report back a summary, and are discarded.

**Tradeoffs.** Hand-off overhead per task — each worker needs its task spelled out completely (which is why the plan contains full code and exact commands rather than intentions). Slightly slower per task, much more scalable across a long project.

**Alternatives: none seriously considered — this was a user directive.** The reasoning above is the justification for the directive, not a comparison against doing all implementation inline; the inline approach was ruled out up front on context-hygiene grounds.

## 2026-07-10 — Portable in-repo context system (the LLM-wiki pattern)

**Problem.** An AI tool needs project context — conventions, current state, the reasons behind past choices. Where does that context live so it survives across tools, machines, and time?

**Options.**
- *Tool-private memory* (Claude's `~/.claude`, or any vendor's built-in memory): convenient and automatic, but it dies with the tool. Switch editors or machines and the context is gone; picking the project up elsewhere means re-explaining it.
- *Markdown in the repo:* `AGENTS.md` (canonical entry point — conventions plus a map of where everything is), `docs/STATE.md` (the handoff: where work stands, what's next, what's blocked), `docs/decisions.md` (this file — the why), `docs/wiki/` (surgical single-topic reference pages). Plain files in git survive tool switches and machine moves. The cost is discipline: a handoff file is only as good as its last update.

**Decision.** Context-as-infrastructure — everything lives as markdown in the repo, the repo is authoritative, and tool-private memory holds only personal preferences. Durability and portability win: picking the project up in a different tool (Gemini CLI, OpenCode) should need zero re-explanation — read AGENTS.md, follow the map, continue. `CLAUDE.md` and `GEMINI.md` are one-line pointers to AGENTS.md since each tool looks for its own filename. Splitting context across small files keeps loading surgical: a worker fixing the HTTP client reads the API wiki page, not the whole history. The discipline cost is paid by a binding convention — update STATE.md after every task — enforced the same way as tests-before-commit.

## 2026-07-10 — Execution as a simulated dev team (PM → leads → workers, council at gates)

**Problem.** Should one AI context do everything — plan, implement, review, talk to the user — or should the work be split across a hierarchy of roles?

**Options.**
- *Flat (PM dispatches everything):* the main context assigns each task and absorbs the results directly. Simple, fewer relay layers, less token overhead. But it teaches little about leading agent teams, and the coordinating context fills with diffs and test output, degrading its judgment over a long project.
- *Simulated dev-shop hierarchy:* the main conversation is PM (scope, priorities, talking to the stakeholder — the user); long-running "tech lead" agents each take a batch of 3–4 plan tasks and run their own workers (one ephemeral implementer per task, one independent reviewer per task); a "council" (five agents with distinct thinking styles, anonymously peer-reviewed then synthesized by a chairman) convenes only at milestone gates. More relay layers, more tokens, and it inherits the known failure modes of human orgs — telephone-game intent drift, rubber-stamp reviewers, leads that quietly rewrite the plan.

**Decision.** The hierarchy, for two reasons honesty requires naming both. First, the stated purpose of this project is learning to lead AI agent teams — a flat model teaches less than a real hierarchy with delegation, escalation paths, and review politics. Second, it protects the expensive resource: the PM context ingests one-minute batch summaries, not diffs and test output, so it stays sharp. The analogy is exact — the same reason ops teams have an incident commander who doesn't touch keyboards. The relay cost is contained by guardrails (leads amortize one context across a whole batch, workers receive verbatim plan tasks with zero discovery, the ~6-agent council is gated to milestones), and `docs/wiki/team-structure.md` lists a specific mitigation for each human-org failure mode. Whether the middle layer earns its cost is itself an experiment: if batch reports show leads adding relay overhead without catching anything the task reviewers miss, we flatten the structure and log that finding here.

## 2026-07-10 — Model tiering across the org chart

**Problem.** Every role in the org chart runs on some model class, and classes differ ~10x in cost. How do you assign them so spend lands where it's justified without starving the roles that decide things? The question is really "which tier per role," and two of those assignments were genuinely contested.

**Options — the contested assignments.**
- *Implementers — Haiku (cheapest), and this is where volume concentrates.* Viable specifically because the plan contains verbatim code and commands, so the work is mechanical transcription, not problem-solving. Putting the volume tier on the cheapest model gives the largest saving. The risk it accepts: a Haiku implementer occasionally burns a round-trip failing before escalation kicks in.
- *Reviewers — Haiku or Sonnet?* The tempting saving was to run reviewers cheap too. Rejected: review is exactly where cheap models fail *quietly*, approving everything, and a rubber-stamp reviewer is worse than none because it launders defects with false confidence. Reviewers stay on Sonnet deliberately.
- *Leads (Sonnet) and PM (Fable/Opus)* were the uncontested ends — coordination and arbitration keep real horsepower; the fewest-tokens, highest-stakes judgment seat keeps the most.

**Decision.** Tier by role — cheapest where volume is highest and the work is mechanical, most expensive where judgment is highest-stakes, and never cheap on review. It's the same discipline as sizing instance types to workloads instead of running everything on the biggest box. A failed cheap attempt plus a Sonnet retry still usually costs less than defaulting everything to Sonnet. Assignments are config, not architecture: when a Haiku implementer fails a task twice the lead escalates to Sonnet and records it, so over time the batch reports map exactly where the cheap tier's ceiling is — precisely the operational knowledge this project exists to produce. If review quality or failure rates say otherwise, the table in `docs/wiki/team-structure.md` changes and this log gets the follow-up.

## 2026-07-10 — Compressed inter-agent communication, uncompressed everything human-facing

**Problem.** Every word an agent writes to another agent is metered twice — once written, once read by the next context. Compressing that traffic saves tokens. What gets compressed, and what must not?

**Options.**
- *Compress everything, uniformly:* maximal saving, but it compresses the wrong things too — a paraphrased work order is a mutated work order (the telephone-game failure mode), and a terse decision log defeats its own educational purpose.
- *Compress only machine-to-machine chatter, with hard exemptions:* status reports, review findings, and handoffs run telegraphic — think syslog lines, not emails: "Task 2: shipped. Review caught missing strict() on schema, fixed. 14 tests pass." Review findings are one line each (location, defect, fix). Two categories stay full-fidelity: *normative content* (plan tasks, requirements, specs — quoted verbatim downward) and *human-facing artifacts* (this log, STATE.md, user reports).

**Decision.** Compress the chatter, exempt the load-bearing categories. The reader of chatter is a machine that doesn't need transitions or politeness, so trimming it is free; but the exemptions are where correctness lives, so they're absolute. The risk is that terse reports hide nuance — a lead writing "shipped" instead of "shipped, but the test only passes because X" loses exactly the signal reviews exist to surface — so report formats mandate specific fields (what review caught, deviations, escalations), and brevity can then only drop padding, not categories. Net savings are real but secondary to the structural levers (batching, verbatim tasks, gated council): this is trimming, not architecture.

## 2026-07-10 — Scope correction: Claude Code is the primary tool; no per-vendor files

**Problem.** The portable-context decision above shipped a per-vendor pointer file (GEMINI.md) alongside CLAUDE.md, on the assumption multiple tools would be in day-to-day use. They aren't — work happens in Claude Code from the terminal. Keep the extra file or drop it?

**Options.** Keep GEMINI.md as ready scaffolding for a second tool, or delete it and add one back only when a second tool actually appears. Keeping it is speculative maintenance (the same YAGNI smell the codebase avoids); deleting it costs a one-line re-add later if ever needed.

**Decision.** Delete GEMINI.md; keep AGENTS.md as the canonical context file with CLAUDE.md importing it. What makes the context system durable was never the pointer files — it's that all project state, decisions, and knowledge live as plain markdown in git, which any future tool can read. If a second tool enters the workflow later, its entry-point file is a one-line add and nothing else changes.

## 2026-07-10 — Deterministic tooling wherever inference isn't judgment

**Problem.** An agent does many kinds of work — choosing goals, computing routes and profit math, reacting to thresholds, building prompt context, recovering from errors. An LLM *can* do all of it, but only some of it *needs* one. Where's the line?

**Options — the taxonomy.** Sorting everything an agent does by whether it truly needs inference (the LLM actually reasoning over a question):
- *Deliberation (choosing goals under ambiguity):* genuinely needs a model.
- *Lookup/arithmetic (routes, prices, profit math), reflexes (refuel below 25%), summarizing (building context), error recovery:* a language model can do these, but each has a right answer a script can compute — an LLM here is a script wearing an expensive costume.

Two candidate mechanisms were also weighed and **deliberately rejected**: auto-repair of invalid plans (guessing intent blurs the validation boundary — we return precise errors instead) and anything social (scripted chat with other players defeats the project's purpose).

**Decision.** Inference is reserved for judgment under ambiguity; everything else is deterministic code. Adopted for Plan 2 as four mechanisms: (1) *reflex policies* — declarative per-agent rules (`keep_fuel_above: 25%`) the executor enforces, waking the planner only when the reflex can't execute, exactly like alert auto-remediation with a pager fallback; (2) a *route macro* (`travel_to(system)`) expanded into jump hops via the game's free routing query; (3) *digest templates* — planner context built as deltas (just what changed) and pre-computed tables, never raw dumps; (4) *market math* in code, handing the planner finished comparisons. Reflexes attack the largest wake category — every `low_fuel` wake was a full planner call to conclude "refuel," a decision with exactly one right answer — and the general effect is that planner wakes shift toward purely strategic, which is what makes the dashboard's wake-reason histogram meaningful. Each reflex is behavior the planner can't override mid-plan, so a badly-tuned one (refueling at a gouging station) repeats its mistake silently; mitigation is that reflexes emit events like everything else, so the dashboard shows their spend.

## 2026-07-10 — Field lesson from Batch A: one branch, one writer at a time

**Problem.** The first executed batch surfaced a classic coordination bug in agent form. While the lead's implementers were committing code tasks, the PM was simultaneously committing documentation to the same branch. An implementer following its task's exact `git add` list swept a file the PM had left staged (a deletion) into a code commit. Nothing broke — but it's the same failure as two admins editing one config file: last writer wins, and the change log lies about who changed what and why. How should concurrent writers be kept from colliding?

**Options.**
- *A lightweight convention:* the PM queues its edits while a batch is in flight and commits only after the batch report arrives; implementers stage exactly their task's listed files and unstage anything else first. Zero new machinery, but it's a discipline that depends on everyone following it.
- *Structural isolation (worktrees — separate working copies of the same repo — or per-role branches with merges):* separate working copies so writers can't collide at all. Buys real isolation, at the cost of merge overhead we don't need at three commits an hour.

**Decision.** The convention, both rules now in the team structure — at three commits an hour the heavier isolation buys isolation we don't need. Evidence it works: the lead added the "stage only your files" instruction mid-batch after the first incident, and it prevented recurrence in the very next two tasks — the delegation layer debugging its own process in real time, precisely the behavior we hoped the hierarchy would show. PM doc updates now lag active batches by up to a batch's duration, which is fine since docs describe decided things and the queue is short. Worktrees are the upgrade path if concurrency grows. (A later decision — protected main — turns this convention into defense in depth rather than the only barrier.)

## 2026-07-10 — Importing a prior project's over-complication lessons

**Concept.** The user's first harness project (modding a simulation game) ran into a wall worth learning from once instead of twice. It submitted a pull request upstream where every bug diagnosis was correct — and the maintainer still rejected the code, personally reimplementing each fix smaller. One ~30-line fix of theirs became 1 line of his, because he fixed where bad data was produced while they had guarded where it crashed; a thread-safety lock they added defended against a caller that, when finally traced, didn't exist. The portable rules distilled from that postmortem now live in `docs/wiki/simplicity-rules.md` and bind all implementation and review work here: fix the producer not the consumer, isolate co-occurring defects before bundling fixes, every new primitive carries a receipt naming the simpler alternative rejected, review verdicts are joint (correct AND smallest — correct-but-larger is a revision), enumerate every input before trusting a cache, and no plan ships with a load-bearing unknown.

**Why this way.** A correct diagnosis wrapped in an oversized fix is still a failed contribution — that's the one-sentence version of the whole postmortem. These rules were battle-tested against a real, skeptical human maintainer, which is better calibration than any self-assessment. And they slot into machinery this project already has: reviewer briefs get the joint verdict rule, the council gate audits suite size and value together, and Plan 2's cache-shaped features (digest templates, reflex policies) must enumerate their inputs per the rule that bit that prior project twice.

**Tradeoffs.** Discipline has a latency cost — stating the invariant before fixing, tracing before locking — but the evidence from that project is that skipping those steps costs more: their reverted hardening commit and evaporated "second bug" were both work that careful sequencing would have avoided. Left behind: the C#/game-specific mechanics (decompilation workflows, IL patching, game-engine thread rules) that don't transfer to a TypeScript service.

## 2026-07-10 — Vendoring private tooling into the repo

**Problem.** The workflow depends on skills and agent definitions that exist only on the user's machine — the council review system, the memory-system conventions, the handoff writer, two prose-quality tools, and specialist agent personas for Docker and security work. On a fresh clone they're missing. How does the tooling travel with the repo?

**Options.**
- *Vendor everything (copy into the repo, the way an application bundles a dependency):* guarantees any clone has what it needs, at the cost that vendored copies are snapshots that drift from the user-level originals.
- *Source-pointer only (catalog what to install):* no drift, but a fresh machine has to fetch each dependency before the workflow runs.

**Decision.** Split by ownership. Private, user-only tooling is vendored under `.claude/skills/` and `.claude/agents/`, where Claude Code discovers it automatically — a few kilobytes of markdown, and the cost of *not* vendoring is discovering on a new machine that the council gate can't run. Public, installable plugins (superpowers, ponytail) are cataloged with their sources instead: copying a maintained plugin freezes its bugs and muddies licensing, so a source pointer is the right weight. The drift risk on vendored copies is accepted with a catalog note to re-sync when either side improves. Full inventory in `docs/wiki/tooling-catalog.md`.

## 2026-07-10 — Protected main: every change lands through a pull request

**Problem.** Main should be protected — no direct commits, everything through a reviewable pull request (PR: a proposed set of changes that gets reviewed, then merged, leaving a permanent record). GitHub enforces this server-side, but branch protection is a paid feature on private repos. How do we enforce it without paying for GitHub Pro?

**Options.**
- *Server-side branch protection (GitHub Pro, or make the repo public):* real enforcement nobody can bypass, but it costs money or forces the repo public before we're ready.
- *A versioned client-side git hook:* a small script in `.githooks/` (a hook is a script git runs automatically at specific moments) that refuses any direct push to main. Free, deterministic, travels with the repo (one `git config core.hooksPath .githooks` per clone). Advisory in the sense that a determined operator can bypass it locally.

**Decision.** The git hook now; server-side protection noted as the option if the project ever goes public. Work happens on short-lived branches — one per batch (`batch/c-store-executor`), one per documentation change (`docs/git-workflow`) — and reaches main only through a PR. Two payoffs: standard change-management hygiene (a PR per batch gives each merge a reviewable summary, a revert point, and an audit trail, the same reason production configs go through merge requests instead of edits on the box), and it structurally fixes the Batch A concurrent-writer problem — a lead on its own branch cannot interleave commits with the PM, so the queue-your-doc-edits convention becomes defense in depth instead of the only barrier. Cost is about a minute of ceremony per batch (branch, PR, merge, pull). The shared working directory still allows one checked-out branch at a time, so batches and PM commits take turns; git worktrees are the upgrade path if true parallelism ever earns its cost.

## 2026-07-10 — Council gate on Plan 1: PASS, with the spike as the real gate

**Concept.** Plan 1's completion went through the project's first council gate: five advisor agents with different thinking styles reviewed the shipped code and tests independently, three peer reviewers scored the anonymized answers (verifying contested claims against source), and a chairman synthesized the verdict. Result: pass. The code is sound, the 63-test suite has no padding to delete, and the one shipped defect (the impossible e2e assertion) was correctly diagnosed and minimally fixed.

**What the gate actually caught.** The process earned its cost in three ways. First, one advisor inverted the "audit for padding" question into "hunt for absence" and found two real coverage holes nobody else saw: the production loop's overlap guard and the retry-exhaustion path are untested. Second, peer review corrected two advisors' wrong (but plausible) account of the Batch D defect mechanism — the concurrency design existed in the plan text before the bad assertion was written; the failure was not tracing it, which changes the fix from "sequence tasks differently" to "sequencing assertions must include their derivation, with file-and-line citations." Third, a reviewer found a compound risk every advisor missed: a failing status query silently disables the low-fuel/low-hull safety wakes (the loop swallows the error and the wake logic skips threshold checks on missing status), which stacks with the response schema's silent-zero fallback — two independent layers of quiet degradation on the same safety path.

**Verdict mechanics and what happens next.** Same-day fixes (three tests, a status_error event, two receipt comments, a fixture freshness check) are dispatched as a follow-up mini-batch. Plan 2's claude-subscription sections stay blocked on the still-unrun container-auth spike — writing assertions against an unverified mechanism would repeat the exact defect class this gate just documented. Plan 2's Ollama and failure-classification sections may be authored now. The trace-before-assert rule becomes a standing plan-authoring instruction.

**Tradeoffs.** The gate cost nine agent runs. Against that: two coverage holes, one corrected root-cause account, one compound safety finding, and a process rule — none of which the per-task reviews caught, because each looked at one diff while the gate looked at the whole. That's the division of labor working as designed.

## 2026-07-10 — Restart resumes silently (documented spec deviation)

**Problem.** The design spec lists a process restart as one of the events that should wake an agent's planner — same logic as any wake: something changed, so the running plan might no longer fit. Should a restart actually re-plan?

**Options.**
- *Wake on restart (what the spec says):* re-validate the plan against the world every time the process comes up. Safest — a plan that went stale during downtime is caught immediately. But every deploy, crash, or container bounce spends one planner invocation to re-approve a plan that's usually still valid — real subscription usage on a decision with no new information, exactly the "reflex, not judgment" case the cost taxonomy says shouldn't reach the LLM.
- *Resume silently (what shipped):* on construction the agent reloads its persisted plan and cursor from SQLite and keeps running, no planner call; `lastPlanAt` is seeded to now so the heartbeat timer doesn't fire next tick either. Cheap, but there's a window (up to one heartbeat) where a restart coincides with a world change that should have interrupted the plan, and the agent won't notice until the next natural wake.

**Decision.** Resume silently — a deliberate, documented deviation from the spec (in `Agent`'s constructor, `src/agent/agent.ts`); the silent resume is by design, not an oversight that happened to also skip the heartbeat. Re-validation isn't skipped forever: the next blocked step or heartbeat routes through the planner as normal, so a genuinely stale plan is caught within one heartbeat window, not never. The risk is bounded by the heartbeat interval, and the resume path is already exercised by the fake-server suite (`test/agent.test.ts`, "cursor persists across restart"). Flagged for Plan 2: the cooldown state machine must revisit restart semantics explicitly — a cooldown mid-wait when the process restarts is a different risk shape than a plan mid-step, and shouldn't inherit this decision by default.

## 2026-07-10 — PR-stage review: conditional, not blanket

**Problem.** With main protected and everything landing via pull request, do PRs get their own review on top of the task-level reviews each change already passed?

**Options.**
- *No PR-stage review:* trust the task-level reviews. Cheapest, but the lead arbitrates its own batch's findings (a mild conflict of interest) and whole-diff interactions go unchecked.
- *Blanket PR review (every PR):* a reviewer on every merge. But by the value-density rule, re-reading already-reviewed diffs mostly produces rubber-stamps, and a rubber-stamp layer is worse than none.
- *Conditional review:* review only PRs that carry new risk — an arbitrated deviation, an escalation, or an integration seam (changes spanning components reviewed in different batches, or safety-path code). Clean code PRs (the lead's report shows zero findings, deviations, escalations) and doc PRs merge on the existing report or PM judgment.

**Decision.** Conditional, because two forces pull opposite ways and only the conditional rule satisfies both. The council gate had just shown that whole-view reviews catch what per-diff reviews structurally cannot (its compound safety finding lived across two files reviewed in different batches), and lead self-arbitration deserves a check exactly when arbitration happened — but blanket review spends a reviewer per merge to mostly confirm work already verified twice. The trigger is mechanical (deviations and escalations are mandatory batch-report fields), so the policy costs no judgment at merge time. Applied retroactively it calibrates right: Batches A–C merge clean, Batch D draws the extra review. First live use (PR #6) behaved as designed — the reviewer probed the one genuine cross-cutting risk (a real-timer test's flakiness potential), ran the suite five times, returned zero manufactured findings. Residual risk accepted: a cross-batch bug in a "clean" batch can still slip to the next council gate.

## 2026-07-10 — Why HTTP instead of MCP (and when MCP comes back)

**Problem.** SpaceMolt markets its MCP endpoint first — MCP being the protocol that describes tools *to a language model* so the model can invoke them directly, schemas and all. For the typical player (a model deciding every action, every tick), MCP is genuinely the most efficient interface. Which interface do we use?

**Options.**
- *MCP:* automatic tool discovery, no glue code, validated calls — ideal when the model is in the action loop. But it puts ~80 tool schemas into every planning prompt, and it assumes the model is the thing making calls.
- *Plain HTTP:* a typed HTTP client the executor (ordinary code) calls, costing zero tokens per call; trivial to fake offline (the zero-token test suite); and the published OpenAPI spec gives us a drift-catching conformance test. Gives up model-driven improvisation.

**Decision.** Plain HTTP, because our architecture puts the model *outside* the action loop — the executor makes nearly every game call, and code doesn't benefit from tool descriptions. MCP optimizes model-driven tool use; our efficiency comes from not driving tools with the model. The planner sees a curated 16-action vocabulary derived from the registry, and tool definitions never enter a prompt at all. What we give up — a model freely exploring the full toolset mid-situation — is real, and is scheduled to be un-made deliberately and boundedly: see the improv-mode entry below.

## 2026-07-11 — The SM-8 experiment: briefing beats model tier

**Problem.** The flight campaign left a question with real money attached: Haiku (the cheap model) flew five sorties without ever selling its cargo, while Sonnet closed the full mine-and-sell loop on its first flight. Why, and what's the fix?

**Options.**
- *Escalate to the bigger model:* assume the gap is capability — Haiku isn't smart enough to close the loop — and promote the miner's replans to Sonnet. Straightforward, but ~10x the subscription burn if it's wrong about the cause.
- *Fix the briefing:* assume the gap is salience, not brains. Audit what Haiku's briefings never contained — a cargo manifest (it saw "19/50 units," never "19x gold ore, sellable here"), goal memory across wakes (each replan started amnesiac), and a runbook-style nudge ("docked at a market with cargo? selling is almost always right") — and add those three lines of deterministic context.

**Decision.** Fix the briefing, and test it: we built the three digest-salience lines, switched the pilot back to Haiku, and relaunched. **Haiku sold within two minutes, then kept selling** — the gap was briefing, not brains. (See the correction entry below: the *intent* finding held, but the sales themselves later proved to be phantoms, so the economic-loop claim was retracted; the tiering lesson survives.) This is the tiering table's first real calibration datum and it points the economical way — a well-briefed cheap model beats a poorly-briefed expensive one at this workload, at roughly a tenth the subscription burn. The general lesson transfers beyond the game: before escalating any agent to a bigger model, audit what its context actually contains; salience failures masquerade as capability failures. (Fittingly, the fixes that made the difference were all deterministic code — the digest builder — not prompt magic.) Sonnet remains one config line away per agent if a harder decision genuinely needs it.

**Tradeoffs.** One datum, one workload, one persona: strategic scatter may still bite on harder decisions (combat, multi-system trade routing), and Sonnet remains one config line away per agent. The experiment protocol itself (annotate the config, bound the trial, define the revert) is now the template for future tuning.

## 2026-07-11 — Standing pilot, invariant promotion, and the PM's own collision

**Concept.** Three process rules formalized at once. First: a test pilot stays online continuously — flight is the training regimen, tuning happens at restarts, and the config annotation records each experiment. Second: invariant promotion, imported from a prior project's change-management discipline — when the same failure class appears twice, it gets promoted from incident note to permanent constraint (deterministic code for pilot failures: the name-to-id normalizer; binding conventions for process failures: the rules below), and the log entry shrinks to a pointer. Third, the newest promotion, earned by the PM personally: **while any worker owns the main working tree, the PM operates server-side only** — merges by PR number touch nothing local; syncing waits. The PM's merge habit (checkout main, pull) yanked the tree out from under a mid-task worker; the worker recovered cleanly and the class joined Batch A's concurrent-writer lesson on its second occurrence, per the promotion rule's own trigger.

**Why this way.** An org that only fixes instances re-fixes them forever; an org that promotes classes gets monotonically harder to break. The evidence base is now long enough to trust: every promoted constraint in this project (staging discipline, PR-only main, dependency-changes-are-tasks, block-reason damping) has held since promotion.

**Tradeoffs.** Constraints accumulate reading cost for every future agent — mitigated by keeping them where they're loaded anyway (AGENTS.md, team-structure.md) and pruning any that stop earning their line.

## 2026-07-11 — Correction to SM-8, and the effect-verification lesson

**Concept.** The SM-8 entry above claimed Haiku "sold within two minutes, then kept selling." Half survives scrutiny. A live state probe (cargo unchanged at 100/100, credits flat at 304) revealed all seventeen "sales" were phantoms: the game accepted each sell call and returned success, but nothing left the hold — this exchange apparently had no demand for those ores — and our executor trusted the response envelope without verifying any state actually changed. ## 2026-07-11 — Batch J: two-compose split, and a secrets-convention mismatch caught before it shipped

**Problem.** Plan 4's single `docker-compose.yml` assumed one deploy target. There are actually two — a staging host (no reverse proxy, since that host doesn't run one, so the dashboard publishes on the staging LAN IP directly for a browser check) and the NAS host (the real promotion, matching the live core stack's verbatim label/network/secrets conventions). How is the compose config structured for both?

**Options.**
- *One file for both:* least duplication, but the targets differ enough (reverse-proxy labels vs a published port, `${SECRETSDIR}` vs a bind path) that a single file needs conditional logic.
- *`docker compose -f base -f override.yml` layering:* the DRY answer — a shared base plus per-target overrides. Real machinery to set up, and it only pays off at three-plus targets.
- *Two standalone files:* `docker-compose.staging.yml` and `docker-compose.prod.yml`, each complete. Duplicates the hardening block (`security_opt`, `cap_drop`, `read_only`, `tmpfs`) across both.

**Decision.** Two standalone files, split before execution and recon-verified against a running service rather than guessed. At two targets the duplication is cheaper than the conditional logic a merged file would need; revisit the `-f base -f override.yml` layering if a third target ever appears. The split immediately earned itself: the NAS host's usual pattern bind-mounts individual secret files read-only to `/run/secrets/<name>` via compose's `secrets:` block, but this app can't use that shape — `ensureCredentials` (`src/config/config.ts:118`) writes a NEW password file per newly-registered agent into its secrets directory at first run, and a read-only single-file mount has no room for a file that doesn't exist yet. Caught before the production file shipped; kept the `${SECRETSDIR}` naming but mounted it read-write as a directory bind, the same shape the staging file already used.

**Blocker hit during execution, not resolved.** The SSH agent (`//./pipe/openssh-ssh-agent`) lists key identities fine but fails at the actual signing step in this non-interactive session ("communication with agent failed") — consistent with the agent requiring an interactive unlock/approval per signing op that has nobody present to grant it. Per `ssh.md`, no bypass was attempted (no raw key file, no `IdentitiesOnly no`). Effect: the real base-image digests, the remote image build, the in-build preflight-gate run, and the staging compose validation could not be executed this session — the Dockerfile ships with `REPLACE_WITH_RESOLVED_DIGEST` placeholders, exactly as the original plan flagged, for the same reason (no reachable Docker daemon), just a different cause this time.

**Problem.** Promoting the dashboard to production means gating it behind the deployment's SSO provider, which needs a proxy provider, an application, and an outpost binding created in the SSO provider. ## 2026-07-11 — The low_fuel runaway: fix the producer, then cap the class

**What happened.** The live miner burned ~75 LLM plan calls an hour for three hours doing nothing. Fuel sat pinned at 32/130, the plan was frozen on its very first step, and the same "I'm low on fuel" wake fired every ~50 seconds — each one throwing away the plan that was already trying to refuel and starting over. This is the exact cost failure the harness exists to prevent (the planner is supposed to run rarely; deterministic code does the work between).

**Why it happened.** The wake that says "fuel is low" checked only the fuel number, not the fact that the in-flight plan already had a refuel step queued up. So it kept interrupting the fix. Underneath sat a second, quieter bug: a one-word typo in the agent's config (`reflexes:` where the code expected `reflex:`) meant the whole reflex block — including the free, no-LLM auto-refuel — was silently dropped and never ran. The config parser accepted unknown keys and discarded them, so the typo was invisible.

**Options — where to fix it.** The simplicity rule ("fix the producer, not the consumer") forces the question of *which* fix, and three were on the table at different locus and cost:
- *Guard the symptom (a hard plan-call cap, only):* cap plan calls per hour and stop the bleeding regardless of cause. Cheapest to write, but on its own it just muzzles a loop that's still wrong — the agent still isn't refueling, just failing quietly instead of expensively.
- *Fix the producer (the wake), only:* make the low-fuel wake stay quiet while the running plan already carries its remedy, so the plan executes instead of being preempted. Addresses the actual cause, but leaves the silent-typo class and any *future* unforeseen runaway unguarded.
- *All three, layered:* producer fix + strict config parsing (unknown keys fail loudly at load) + a cause-agnostic hard ceiling as a backstop.

**The fix, in layers.** All three, because each buys a different guarantee and the cheap ones aren't substitutes for the real one:
- Producer fix: the low-fuel wake now stays quiet while the running plan already carries its remedy. The plan gets to execute instead of being preempted. This is "fix the cause, not the symptom" — the wake was the producer of the bad loop, so this removes *this* cause.
- Loud config: the parser is now strict. An unknown or misspelled key fails at startup instead of vanishing. The class of "silent typo disables a safety feature" is closed, not just this one instance.
- Hard ceiling: independent of any root cause, each agent can now make at most N plan calls per rolling hour (default 12). If a future bug we didn't foresee tries to run away, the cost is capped and the agent falls back to deterministic reflexes. Operator instructions bypass the cap — the human is always able to steer.

A hard cap *alone* would have been guarding the consumer — treating the symptom — which the simplicity rule exists to forbid; but discarding the cap because we'd fixed the cause would throw away the only defense against the *next* unknown bug. That's why the layers aren't redundant.

**The result, honestly.** After deploy: zero low-fuel wakes (was 231 of 233). The runaway is dead. But killing it revealed the next layer — with fuel no longer storming, the pilot now trips over the game's timing: it issues an action before the ~10-second game tick has finished the previous one, gets told "another action is already in progress," and replans instead of waiting. It is the same shape of mistake one level up (reacting to a condition that resolves itself if you just wait), and it is now bounded by the ceiling rather than a runaway. That is the next producer fix.

**The lesson worth keeping.** This whole episode is the whitepaper's line made concrete: "most agent failures are configuration failures." The model was never the problem. A typo and a missing guard were. The guardrails now make that class loud and bounded — which is what an engineered harness, as opposed to a vibe-coded script, is for.

## 2026-07-11 — Transient-block thrash (SM-10/SM-11): wait for the tick, don't replan

**What happened.** Killing the low_fuel runaway uncovered the next layer. The game resolves actions on a ~10-second tick; a ship that just jumped or started travelling is briefly "in transit," and any command issued in that window comes back blocked with "wait for the jump/travel to complete, then resubmit this command." The executor treated that transient, self-resolving block exactly like a permanent one (can't-afford, no-route) and replanned — calling the LLM — instead of simply waiting a beat. Same shape of mistake as the low_fuel bug, one level up: reacting to a condition that fixes itself if you wait.

**Options — how to tell a transient block from a terminal one.**
- *Enumerate the transient phrases:* match the specific block strings we'd seen ("mid-travel", "action already in progress") and wait on those. Simple and quick to ship — but it depends on having seen every phrase the game can emit.
- *Key on an authoritative state flag:* hold any action while the ship's own `in_transit` flag is set, regardless of what prose the block carries. Robust to unseen phrasing, if the flag is reliably set.
- *Key on the game's canonical instruction:* where prose is still needed, match the game's own retry signal — "resubmit this command" — which only appears when the game is telling you to retry the same action.

**Two passes, because the field taught us the general case.** The specific-phrase approach shipped first (SM-10) and the field immediately falsified it: the pilot hit "mid-**jump**" — a phrase we hadn't enumerated — on a mine step, which slipped through and replanned until the plan-rate ceiling capped it. That miss was the useful signal: enumerating the game's error prose is a losing game. The second fix (SM-11) generalised both robust ways at once: it holds ANY action while the ship's authoritative `in_transit` flag is set (a mine issued mid-jump is now held by the flag, not by string-matching), and where prose is still needed it keys on the game's own canonical instruction — "resubmit this command" — which only appears when the game is telling you to retry the same action. Terminal blocks (do something different) still replan.

**The defense-in-depth worked as designed.** The first fix had a gap; an independent review had explicitly predicted that class ("a novel transient phrasing would replan, bounded by the heartbeat backstop"). When the gap hit in the field, the plan-rate ceiling — the blunt, cause-agnostic backstop we built precisely for unforeseen runaways — capped the cost while we widened the fix. The precise guard missed; the crude cap held. That is the point of layering them.

**Result and honesty.** Field-confirmed after deploy: the executor now emits "wait" and holds through transit instead of replanning; the pilot jumps and travels across systems cleanly; plan-rate dropped from ~75/hr to ~24/hr and is bounded. Two assumptions remain ASSUMED, not verified, for lack of a live capture: that `in_transit` is truly set during an inter-system jump (the prose fallback covers us if it isn't), and that terminal blocks never emit "resubmit this command" (a false match would be caught by the heartbeat wake). Both are tagged in STATE and will be confirmed or corrected in a live window, not left as silent load-bearing guesses.

**What's next, and why it's a different kind of problem.** With the executor no longer thrashing, the remaining inefficiency is planning quality: the pilot wanders between systems and tries to mine or dock where it can't. That is a briefing/salience problem (the SM-8 class), not a guardrail one — and the right move is to build the measurement first. Batch 2 (no-progress detection plus a stats/trends dashboard) is the trajectory-eval instrumentation that lets us diagnose the wandering with data instead of anecdote, exactly as the SDLC practices now prescribe: set the bar at the eval, not the demo.
