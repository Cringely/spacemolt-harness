**Improv-mode Batch 0 (live MCP probe) DONE — all three load-bearing unknowns resolved (2026-07-12, branch `batch-0-results`).** Before writing any improv-mode transport code we ran a narrow, authorized, read-only live MCP probe against our own account and captured the scrubbed fixture at `test/fixtures/mcp-probe-2026-07-12.json`. **Action RESULTS on the dashboard shipped (2026-07-12, `feat/action-results-observability`, PR open).** The feed showed what pilots DID, not what they GOT. Two new report-only observers, both pure retain-and-expose over data the loop already fetches each tick (zero extra game call, zero LLM): (1) `notification` events — the game's OWN result feed (mining yields, sales, combat) surfaced from the notifications the loop already polls, deduped by id (bounded set, cap 500) so a repeated poll doesn't restamp the feed; (2) `ledger` events — a per-tick DELTA of status (a cargo item's quantity rising "+3 Carbon Ore, cargo 8->11", credits moving "+4cr"/"-186cr"), emitted only when something changed (silent on a quiet tick). The ledger is the reliable answer to "how much ore this tick, which resource, sell revenue" precisely because `mine`'s yield resolves a tick LATER in status, not in the action result. Dashboard: action-result display widened 80→160 chars; `notification` + `ledger` added to the feed filter KNOWN_TYPES, default-ON (low-volume, unlike status_snapshot). Deliberately a DELTA, never absolutes — status_snapshot carries the absolute series (hidden by default), so no double-count. Decision-log entry (A emit-notifications+delta-ledger vs B parse-result-strings vs C cumulative-stats-panel; A wins because it's the one source that sees the async mine yield) + improv-mode §5 backstop note added. New ablation tests (dedupe holds across repeated polls; cargo-rise names the resource+delta; credit change carries the sign; no-change tick stays silent). 397 offline tests green, typecheck clean.
